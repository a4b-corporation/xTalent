## Overview

Statutory Rules are versioned records that encode government-mandated rates, ceilings, and brackets used in payroll calculations. Examples include BHXH/BHYT/BHTN contribution rates, PIT income brackets, minimum wages by region, and OT multipliers. Each rule version has a validity period (`valid_from` / `valid_to`). A new government circular creates a new rule version — the old version remains active until superseded.

Rules with `legal_entity_id = NULL` are global (apply to all legal entities). Rules with a specific `legal_entity_id` override the global rule for that entity.

## Business Rules

- **BR-020–035** (SI): SI rates are defined as SI_RATE rules. Employer and employee rates are stored separately per fund (BHXH, BHYT, BHTN). Ceilings (monthly SI salary cap) are also stored as rule attributes.
- **BR-040–051** (PIT): PIT_BRACKET rules store a bracket table (income floor, income ceiling, rate percentage, quick deduction amount).
- **BR-070–073** (Minimum wage): MINIMUM_WAGE rules store region code → amount mappings.
- Once a rule is ACTIVE, it cannot be edited. To change an active rule, a new version must be created.
- Rule versions for the same `rule_type + legal_entity_id` combination must not have overlapping `valid_from / valid_to` date ranges. The system rejects overlapping activation.
- Activation requires PLATFORM_ADMIN role. Creating a rule version (DRAFT) is also restricted to PLATFORM_ADMIN.
- A rule used in a LOCKED period result is permanently referenced by that result's calc log. Deleting or deactivating it does not break historical records.

## CRUD Specification

### Fields — Core (all rule types)

| Field | Type | Required | Validation | Notes |
|-------|------|----------|------------|-------|
| rule_type | Enum | Yes | SI_RATE / PIT_BRACKET / MINIMUM_WAGE / OT_MULTIPLIER / LUNCH_ALLOWANCE_CAP | Controls payload structure (dynamic section) |
| rule_name | String | Yes | Max 200 chars | e.g., "BHXH/BHYT/BHTN 2024 — Circular 10/2024/TT-BLDTBXH" |
| legal_entity_id | UUID | No | NULL = global; must exist if provided | Null = applies to all entities |
| valid_from | Date | Yes | |
| valid_to | Date | No | Must be > valid_from if provided. Null = open-ended until superseded |
| reference_document | String | No | Max 500 chars | Circular or decree number for audit reference |
| status | Enum | System | DRAFT / PENDING_APPROVAL / ACTIVE / SUPERSEDED | |
| notes | String | No | Max 1000 chars | Internal notes |

### Dynamic Payload Sections per rule_type

#### rule_type = SI_RATE

Employer contribution rates (% of SI_BASIS):

| Field | Type | Notes |
|-------|------|-------|
| bhxh_employer_rate | Decimal | e.g., 0.175 = 17.5% |
| bhyt_employer_rate | Decimal | e.g., 0.030 = 3.0% |
| bhtn_employer_rate | Decimal | e.g., 0.010 = 1.0% |

Employee contribution rates:

| Field | Type | Notes |
|-------|------|-------|
| bhxh_employee_rate | Decimal | e.g., 0.080 = 8.0% |
| bhyt_employee_rate | Decimal | e.g., 0.015 = 1.5% |
| bhtn_employee_rate | Decimal | e.g., 0.010 = 1.0% |

Ceilings:

| Field | Type | Notes |
|-------|------|-------|
| si_monthly_ceiling | Decimal | Monthly SI salary ceiling (VND) for BHXH/BHTN |
| bhyt_monthly_ceiling | Decimal | Monthly SI salary ceiling (VND) for BHYT (may differ) |

UI: all fields rendered as labeled decimal inputs. Rates displayed as percentage format (e.g., 17.5%) but stored as decimals (0.175).

#### rule_type = PIT_BRACKET

Bracket table (progressive income tax). Each row:

| Column | Type | Notes |
|--------|------|-------|
| bracket_number | Integer | Display order (1, 2, 3…) |
| income_floor | Decimal | Lower bound of bracket (VND/year or VND/month per local law) |
| income_ceiling | Decimal | Upper bound (NULL for top bracket) |
| tax_rate | Decimal | e.g., 0.05 = 5% |
| quick_deduction | Decimal | Preset deduction amount for quick calculation method |

UI: inline editable table. [Add Bracket] / [Remove Bracket] buttons. System validates: brackets must be contiguous (no gaps, no overlaps), ordered ascending by income_floor.

#### rule_type = MINIMUM_WAGE

Region-to-amount mapping. Each row:

| Column | Type | Notes |
|--------|------|-------|
| region_code | Enum | REGION_1 / REGION_2 / REGION_3 / REGION_4 (Vietnam) |
| monthly_minimum | Decimal | VND per month |
| hourly_minimum | Decimal | VND per hour (derived from monthly / 26 / 8 but stored explicitly) |

UI: four-row table (one row per region), always exactly four rows for VN entity rules.

#### rule_type = OT_MULTIPLIER

| Field | Type | Notes |
|-------|------|-------|
| weekday_rate | Decimal | e.g., 1.5 = 150% of base hourly rate |
| weekend_rate | Decimal | e.g., 2.0 = 200% |
| holiday_rate | Decimal | e.g., 3.0 = 300% |
| night_ot_supplement | Decimal | Additional multiplier for night OT (e.g., 0.3) |

#### rule_type = LUNCH_ALLOWANCE_CAP

| Field | Type | Notes |
|-------|------|-------|
| monthly_cap | Decimal | Maximum tax-exempt lunch allowance per month (VND) |

## Timeline View

**Primary navigation pattern for statutory rules.**

Located at: Payroll > Statutory Rules > [Select rule_type]

Renders a horizontal timeline per rule_type showing all versions:

```
[SUPERSEDED v1]──────[ACTIVE v2]──────────[DRAFT v3 (upcoming)]
2020-01-01          2024-01-01             2025-07-01
```

- Each segment is colored by status:
  - SUPERSEDED: gray
  - ACTIVE: green
  - DRAFT / PENDING_APPROVAL: blue (dashed border to indicate not yet in effect)
- Clicking a segment opens the full rule version detail panel (read-only if ACTIVE or SUPERSEDED).
- [+ Create New Version] button on the timeline header.
- If a DRAFT or PENDING_APPROVAL version exists for a rule_type, the [+ Create New Version] button is disabled with tooltip: "A pending version already exists. Activate or discard it before creating another."

## Activation Workflow

```
DRAFT → [Submit] → PENDING_APPROVAL → [Activate] → ACTIVE
                            ↓
                     [Reject/Discard] → DRAFT (editable again)
```

- **DRAFT**: created by PLATFORM_ADMIN; editable. Submit sends notification to other PLATFORM_ADMINs for second-eye check.
- **PENDING_APPROVAL**: read-only. A different PLATFORM_ADMIN (not the creator) can activate or reject.
- **ACTIVE**: read-only. Cannot be edited. Can be superseded by activating a new version.
- **SUPERSEDED**: read-only archive. Retained indefinitely for audit and historical calc log references.

## Error Handling

| Scenario | Behavior |
|----------|---------|
| Overlapping valid_from / valid_to with existing ACTIVE version | Activation rejected: "Date range overlaps with active version [rule_name] valid [start]–[end]. Adjust valid_from or end the existing version first." |
| Creator attempts to activate own submission | Activation blocked: "Self-approval not permitted. A different PLATFORM_ADMIN must activate this version." |
| PIT bracket table has gaps or overlaps | Save rejected with inline table errors: "Gap detected between bracket 2 and bracket 3." |
| Rule referenced in a LOCKED period | Deactivation allowed but shown with warning: "This rule version was used in [N] locked periods. Historical calc logs remain valid." |
