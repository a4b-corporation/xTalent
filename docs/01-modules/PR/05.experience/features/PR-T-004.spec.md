# PR-T-004: Period Lock

**Type**: T (Transaction)
**Priority**: P0
**BC**: BC-03 — Payroll Execution Engine
**Actors**: PAYROLL_ADMIN (initiates), FINANCE_MANAGER (must have approved first — pre-condition)
**Linked Stories**: US-028, US-029
**API Endpoints**:
- `GET /api/v1/payroll-usecase/pay-periods/{period_id}/lock-readiness` — Check lock pre-conditions
- `POST /api/v1/payroll-usecase/pay-periods/{period_id}/lock` — Execute lock
- `GET /api/v1/payroll-usecase/pay-periods/{period_id}/integrity` — Get integrity hash results

---

## Overview

Period Lock is the final irreversible action in the payroll cycle. Once a period is locked, all worker results become immutable. The lock also triggers SHA-256 hash generation per worker result record, which is used for future integrity verification (PR-A-004). Because this action cannot be undone, it requires explicit confirmation from the Payroll Admin and a verified set of pre-conditions.

---

## Pre-Conditions

Before the [Lock Period] button is rendered as active, the system checks:

| Pre-Condition | Check |
|---------------|-------|
| Payroll run status | Run for this period must be APPROVED |
| Open exceptions | Zero open exceptions on the approved run |
| Bank payment file | File must have been generated (status: GENERATED or SENT) OR admin has explicitly waived this step |
| Payslip generation | Not a hard blocker in V1; shown as advisory warning if payslips not yet generated |

**Pre-condition check display**: The lock readiness panel shows each condition as a checklist with status (green checkmark / red X / amber warning). The [Lock Period] button is only enabled when all hard pre-conditions pass.

**Bank payment waiver**: If the bank file has not been generated, the system shows:
"Bank payment file has not been generated for this period. You may proceed without it by checking this box: [ ] I confirm bank payment will be handled outside xTalent for this period."
Checking the box satisfies the bank payment pre-condition and adds a waiver note to the lock record.

## Lock Confirmation Modal

Triggered by clicking [Lock Period].

The modal has two sections:

### Section 1: Period Summary

Read-only summary confirming what is being locked:

| Field | Value |
|-------|-------|
| Pay Group | [name] |
| Period | [start] – [end] |
| Approved Run ID | [run_id] |
| Total Workers | [N] |
| Total Net Pay | [amount] VND |
| Approved by (Level 2) | [HR Manager name] at [timestamp] |
| Approved by (Level 3) | [Finance Manager name] at [timestamp] |

### Section 2: Typed Confirmation

Warning text:
"This action is IRREVERSIBLE. Once locked, this period's payroll results cannot be modified. A new correction run must be initiated for any subsequent changes."

Confirmation input:
"Type **LOCK** to confirm:"
[ text input field ]

[Lock Period] button in modal is enabled only when the input exactly matches "LOCK" (case-sensitive).

## Post-Lock Behavior

On successful lock:

1. Period status → LOCKED.
2. System initiates async SHA-256 hash generation for all worker results in the period.
3. A PERIOD_LOCKED audit event is written.
4. Toast: "Period [Period Name] is now locked."

**On the period detail page after lock**:

- Status badge: LOCKED (displayed in a distinct locked style — padlock icon + "LOCKED" text)
- All action buttons disabled (no Submit, no Approve, no Edit)
- Integrity hash section appears:
  - Hashing status: GENERATING → COMPLETE
  - Summary: [N] hashes generated, [N] verified, [N] failed (if any)
  - [View Integrity Report] link → PR-A-004

**Downstream triggers**:
- Bank payment file generation becomes available (if not already done)
- Payslip generation becomes available
- GL journal generation becomes available
- These are independent follow-on actions (PR-T-006, PR-T-007, PR-T-019)

---

## Integrity Hash Details

Per worker result, the system computes:

```
SHA-256(
  run_id + working_relationship_id + period_id +
  gross_amount + si_employee_total + pit_withheld + net_amount +
  locked_at_timestamp
)
```

The hash is stored in `payroll_result.integrity_hash`.

Post-lock, the integrity section on the period detail shows:
- Hashing job status (RUNNING / COMPLETE / FAILED)
- On COMPLETE: count of hashed records vs expected
- On any FAILED hashes: alert banner with severity HIGH

## Business Rules

| Rule | Description |
|------|-------------|
| BR-096 | SHA-256 integrity hash generated for every worker result at lock time |
| BR-097 | A locked period cannot be modified. No further pay_result writes are accepted |
| BR-082 | Run must be in APPROVED status before lock is permitted |

## Error Handling

| Scenario | Behavior |
|----------|---------|
| Lock attempted with open exceptions | Pre-condition check fails; [Lock Period] button disabled; message: "N open exceptions must be acknowledged before locking." |
| Lock attempted before run is APPROVED | Pre-condition check fails; message: "Run [run_id] is in [current status]. Full approval is required before locking." |
| Lock attempted by non-PAYROLL_ADMIN | HTTP 403; action not rendered in UI |
| Typed confirmation mismatch | [Lock Period] button in modal remains disabled until exact "LOCK" is typed |
| Hash generation fails post-lock | Alert on period detail: "Integrity hash generation failed for [N] records. Contact Platform Admin." Link to PR-A-004. |
| Double-lock (already locked) | HTTP 409; UI never shows the Lock button for locked periods |
