# PR-T-001: Monthly Payroll Run

**Type**: T (Transaction)
**Priority**: P0
**BC**: BC-03 — Payroll Execution Engine
**Actors**: PAYROLL_ADMIN (primary executor)
**Linked Stories**: US-010, US-011, US-012, US-013, US-014, US-015, US-016, US-017, US-018, US-019
**API Endpoints**:
- `POST /api/v1/payroll-usecase/payroll-runs` — Initiate run
- `GET /api/v1/payroll-usecase/payroll-runs/{run_id}` — Get run detail and status
- `GET /api/v1/payroll-usecase/payroll-runs/{run_id}/progress` — Progress (WebSocket + polling fallback)
- `POST /api/v1/payroll-usecase/payroll-runs/{run_id}/override-ta` — Override TA data (MANUAL_ATTENDANCE_OVERRIDE)
- `POST /api/v1/payroll-usecase/payroll-runs/{run_id}/submit-for-approval` — Submit after all exceptions acknowledged
- `POST /api/v1/payroll-usecase/payroll-runs/{run_id}/retry` — Retry a FAILED run

---

## Overview

The Monthly Payroll Run is the central transaction in the payroll module. It computes gross-to-net pay for every worker enrolled in a Pay Group for a given period. The run is multi-step, asynchronous, and stateful: the admin guides it through a wizard while the computation engine works in the background. The UI persists run state server-side so the admin can navigate away and return at any time.

This feature covers the full run lifecycle from setup through submission for approval. Approval is covered in PR-T-003. Exception handling is detailed in PR-T-002 (dedicated screen), though exceptions are also surfaced within this wizard.

---

## Run State Machine

```
QUEUED
  → PRE_VALIDATING     (system checking data completeness)
  → RUNNING            (engine computing worker results)
  → COMPLETED          (all workers computed; may have exceptions)
  → PENDING_APPROVAL   (submitted; awaiting HR + Finance sign-off)
  → APPROVED           (both approval levels granted)
  → LOCKED             (period locked; immutable)
  → FAILED             (run error; re-initiable)
```

Transitions:
- QUEUED → PRE_VALIDATING: automatic on creation
- PRE_VALIDATING → RUNNING: after admin confirms (override or accept pre-validation results)
- RUNNING → COMPLETED: automatic when engine finishes all workers
- COMPLETED → PENDING_APPROVAL: admin action (Submit for Approval); only if open exceptions = 0
- PENDING_APPROVAL → APPROVED: via PR-T-003
- APPROVED → LOCKED: via PR-T-004
- PRE_VALIDATING / RUNNING / COMPLETED → FAILED: system on engine error
- FAILED → QUEUED: admin action (Retry — creates new run_id)

## Wizard Steps

### Step 1: Setup

**Purpose**: Configure the run parameters before execution.

Fields:

| Field | Type | Required | Notes |
|-------|------|----------|-------|
| pay_group_id | Select | Yes | Only ACTIVE pay groups for the admin's legal entity |
| period | Select | Yes | Only OPEN periods for the selected pay group; default = current open period |
| run_mode | Radio | Yes | DRY_RUN / SIMULATION / PRODUCTION — see modes below |
| statutory_rule_effective_date | Date | Yes | Defaults to period end date; used to select applicable statutory rule versions |

**Run Modes**:
- **DRY_RUN**: Full computation but results are not persisted. For testing formula changes. No exceptions raised. Cannot be submitted for approval.
- **SIMULATION**: Full computation, results persisted with `is_simulation = true`. Exceptions visible. Cannot be submitted for approval.
- **PRODUCTION**: Full computation, results persisted as final. Exceptions must be resolved. Submittable for approval.

After selecting pay_group + period, the system shows a read-only pre-flight summary:
- Workers enrolled: [N]
- Period dates: [start] – [end]
- Pay calendar cut-off: [date]
- Prior run in this period: [None / Simulation run on date]
- Applicable statutory rules as of [statutory_rule_effective_date]: list of rule_name + valid_from

[Start Run] button initiates the run and transitions to Step 2.

---

### Step 2: Pre-Validation

**Purpose**: Surface data completeness issues before the engine runs, so the admin can address them or override.

The system automatically performs checks on entering this step. Progress indicator shows "Running pre-validation checks..."

**Checklist items** (each shows PASS / WARN / FAIL):

| Check | Description |
|-------|-------------|
| TA data completeness | Time & Attendance data exists for all workers for the full period |
| SI eligibility matrix | All workers have valid contract_type → SI eligibility mapping |
| Bank account coverage | All workers have at least one active bank account |
| Pay profile binding | All workers have an active pay profile for this period |
| Statutory rules available | SI_RATE, PIT_BRACKET, MINIMUM_WAGE rules exist for the period's effective date |

Display: table of checks with status badge (green PASS / yellow WARN / red FAIL), check name, and detail message for WARN/FAIL.

**Actions per result**:
- All PASS: [Proceed to Run] button enabled.
- WARN items: admin can proceed with warnings (acknowledged in the run record).
- FAIL items: admin must resolve or override:
  - TA data: [Enter Manual Attendance] opens PR-T-002 exception flow variant: admin enters total work days / hours manually for flagged workers. This creates a MANUAL_ATTENDANCE_OVERRIDE record.
  - Other FAIL: admin must fix in the source system (CO or TA module) and re-run pre-validation.

[Re-run Checks] button available after admin makes changes.
[Proceed] button enabled when all FAILs are resolved or explicitly overridden.

---

### Step 3: Running (Async)

**Purpose**: Monitor live computation progress.

Display while engine is running:

- Progress bar: workers computed / total workers (e.g., "234 / 1,200 workers")
- Estimated time remaining (derived from per-worker average so far)
- Run mode badge (DRY_RUN / SIMULATION / PRODUCTION)
- Current run_id (for support reference)
- [Cancel Run] button — admin can cancel a running PRODUCTION run; status → FAILED; a new run can be initiated.

**Connectivity**:
- Primary: WebSocket subscription to `run/{run_id}/progress` events
- Fallback: 5-second polling of `GET /payroll-runs/{run_id}/progress` if WebSocket is unavailable
- Admin can navigate away. On return, the current step is restored from run state. A banner reads: "Payroll run in progress. Last update: [timestamp]."

**On completion**:
- Automatic transition to Step 4 (Exceptions) if any exceptions exist.
- If no exceptions: automatic transition to Step 5 (Review Summary).
- Toast notification: "Run completed. N workers processed."

---

### Step 4: Exceptions

**Purpose**: Review and acknowledge all exceptions before proceeding.

This step is skipped if no exceptions were raised (run transitions directly to Step 5).

Displays the exception list. Detailed exception resolution is specified in PR-T-002. Within this wizard step, the display is a condensed version:

| Column | Notes |
|--------|-------|
| Exception Type | Colored badge |
| Worker Name | |
| Exception Detail | One-line summary |
| Status | OPEN / ACKNOWLEDGED |
| Action | Acknowledge (inline) or [Manage Exceptions] deep link to PR-T-002 screen |

Summary banner: "N open exceptions. All exceptions must be acknowledged before submission."

Bulk acknowledge: [Acknowledge All of Same Type] — selects all OPEN exceptions of the same type; admin enters one reason_code for the batch.

[Refresh] button re-checks exception status.

The [Proceed to Review] button is enabled only when open exception count = 0.

---

### Step 5: Review Summary

**Purpose**: Confirm run totals before submitting for approval.

Summary cards (large format):

| Card | Value |
|------|-------|
| Total Gross | Sum of gross_amount across all workers |
| Total SI (Employee) | Sum of si_employee_total |
| Total SI (Employer) | Sum of si_employer_total |
| Total PIT Withheld | Sum of pit_withheld |
| Total Net Pay | Sum of net_amount |
| Workers Included | Count of workers with net_amount > 0 |
| Workers with Zero Net | Count of workers with net_amount = 0 |

**Period-over-period comparison** (shown when a prior PRODUCTION run exists for the same pay group):

| Metric | Current | Previous | Variance Amount | Variance % |
|--------|---------|----------|----------------|------------|
| Gross | | | | |
| Net | | | | |
| Headcount | | | | |

Rows where variance % exceeds the pay group's `variance_alert_threshold` are highlighted in amber.

Download options: [Download Summary PDF], [Download Worker Detail CSV].

[Submit for Approval] button:
- Enabled only for PRODUCTION runs with 0 open exceptions.
- Confirmation modal: "This will submit the run for HR Manager and Finance Manager approval. Once submitted, results cannot be changed without rejection and re-submission."
- On confirm: run status → PENDING_APPROVAL; notifications sent to approvers; admin sees confirmation screen with run_id and PENDING_APPROVAL badge.

## Run Detail Page

Accessible from: Payroll > Payroll Runs > [Run ID]

The run detail page is the persistent home for all run-related activity. It shows:

**Header**: Run ID, Pay Group, Period, Run Mode, Status badge, Initiated by, Initiated at.

**Summary cards**: same as Step 5 (updated when run is complete).

**Exceptions badge**: count of open/total exceptions; links to PR-T-002 exceptions screen.

**Approval status** (when PENDING_APPROVAL or beyond): two-level approval tracker showing each approver's status (PENDING / APPROVED / REJECTED) and timestamp.

**Worker Result Table**:
- Columns: Worker Name, Employee ID, Gross, SI (Employee), PIT, Net, Exceptions count, Calc Log
- Paginated: 50 rows per page
- Search: by worker name or employee ID
- Filter: by exception status (Has Exceptions / No Exceptions)
- [View Calc Log] link per row (opens PR-A-006)

---

## Re-Run Capability

When a run has status = FAILED:

- Banner on run detail: "This run failed. Reason: [error message]."
- [Retry Run] button: creates a new PayrollRun record with a new run_id. The wizard starts from Step 1 with prior parameters pre-filled.
- The failed run record is retained in the run history with status = FAILED.

## Business Rules Enforced

| Rule | Description |
|------|-------------|
| BR-080 | A pay period must have exactly one PRODUCTION run in COMPLETED or later state |
| BR-081 | DRY_RUN results are never persisted to payroll_result table |
| BR-082 | PRODUCTION runs require two-level approval before lock |
| BR-083 | Finance Manager (level 3) cannot approve before HR Manager (level 2) |
| BR-090 | Negative net pay raises NEGATIVE_NET exception; not auto-zeroed |
| BR-091 | Gross below minimum wage raises MIN_WAGE_VIOLATION exception |
| BR-092 | OT hours exceeding cap raises OT_CAP_EXCEEDED exception |
| BR-094 | SI-eligible worker with zero SI basis raises ZERO_BHXH_SI_ELIGIBLE exception |
| BR-095 | Calc log must be complete for all FINAL worker results |
| BR-096 | Integrity hash generated at period lock |

## Error Handling

| Scenario | Behavior |
|----------|---------|
| Engine error mid-run | Run status → FAILED; admin notified in-app and email; error detail shown on run detail page |
| Network disconnect during running step | WebSocket reconnects automatically; falls back to polling; progress bar resumes from last known state |
| User navigates away during running step | Progress continues server-side; banner on other pages: "A payroll run is in progress for [Pay Group] [Period]" |
| Submit attempted with open exceptions | [Submit for Approval] button disabled; tooltip: "N open exceptions must be acknowledged first." |
| Concurrent run initiation (race condition) | Second initiation rejected: "A run for [Pay Group] [Period] already exists (run_id: [X]). Cancel the existing run before initiating a new one." |
