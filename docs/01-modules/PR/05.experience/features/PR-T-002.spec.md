# PR-T-002: Payroll Exception Handling

**Type**: T (Transaction)
**Priority**: P0
**BC**: BC-03 — Payroll Execution Engine
**Actors**: PAYROLL_ADMIN (primary), FINANCE_MANAGER (co-approval for SET_TO_ZERO resolution)
**Linked Stories**: US-016, US-017
**API Endpoints**:
- `GET /api/v1/payroll-usecase/payroll-runs/{run_id}/exceptions` — List exceptions for a run
- `POST /api/v1/payroll-usecase/payroll-runs/{run_id}/exceptions/{exception_id}/acknowledge` — Acknowledge single exception
- `POST /api/v1/payroll-usecase/payroll-runs/{run_id}/exceptions/bulk-acknowledge` — Bulk acknowledge
- `GET /api/v1/payroll-usecase/payroll-runs/{run_id}/exceptions/export` — Export as CSV

---

## Overview

When a payroll run completes, exceptions are raised for worker results that violate statutory or business rules. Exceptions are not blocking errors — the calculation completes, but the result is flagged for human review. Every exception must be acknowledged (with a reason code and optional note) before the run can be submitted for approval.

The Exceptions screen is accessible both inline from the PR-T-001 wizard (Step 4) and as a standalone screen reachable from the run detail page. The standalone screen supports bulk operations that the inline wizard step does not.

## Exception Types and Resolution Actions

### NEGATIVE_NET

**Condition**: Calculated net pay for a worker is negative (total deductions exceed gross).

**Typical causes**: Large ad-hoc deduction in the same period as a low-attendance month; SI deduction applied to a partial-period worker.

**Resolution options** (admin must choose one):

| Option | Code | Description | Additional Approval |
|--------|------|-------------|---------------------|
| Zero net with recovery | ZERO_NET_WITH_RECOVERY | Set net to 0 this period; recover the negative delta from the next period's net pay | None |
| Set to zero (write-off) | SET_TO_ZERO | Set net to 0; the negative delta is absorbed (not recovered) | Finance Manager co-approval required |

If SET_TO_ZERO is chosen: after the admin acknowledges, the exception status becomes PENDING_FINANCE_APPROVAL. The Finance Manager receives an in-app notification. The Finance Manager must confirm or reject on the exception detail. Only after Finance Manager confirmation is the exception status ACKNOWLEDGED.

---

### MIN_WAGE_VIOLATION

**Condition**: Calculated gross pay for a worker falls below the applicable minimum wage for the worker's region.

**Resolution options**:

| Option | Code | Description |
|--------|------|-------------|
| Adjust to minimum | ADJUST_TO_MINIMUM | System top-up the gross to minimum wage level automatically |
| Acknowledge with reason | ACKNOWLEDGE_WITH_REASON | Confirm violation is known and intentional (e.g., part-time, trainee exemption); requires free-text reason |

---

### OT_CAP_EXCEEDED

**Condition**: Overtime hours for a worker exceed the monthly OT cap defined on their pay group or pay profile.

**Resolution options** (admin must choose a reason code from an enum):

| Code | Meaning |
|------|---------|
| MANAGEMENT_APPROVAL | Overtime approved by management for business continuity |
| EMERGENCY | Emergency situation (natural disaster, equipment failure) |
| PROJECT_DEADLINE | Critical project deadline approved in advance |

A free-text note is required in addition to the reason code.

---

### ZERO_BHXH_SI_ELIGIBLE

**Condition**: A worker is classified as SI-eligible based on their contract_type, but their calculated SI_BASIS is zero.

**Typical causes**: Worker had full-month unpaid leave; data mapping error between contract_type and eligibility matrix.

**Resolution options**:

| Option | Code | Description |
|--------|------|-------------|
| Force include | FORCE_INCLUDE | Override the zero SI_BASIS; admin must manually enter a basis amount |
| Confirm exempt | CONFIRM_EXEMPT | Worker is correctly exempt this period (e.g., approved unpaid leave); log reason |

When FORCE_INCLUDE is chosen: a field appears for the admin to enter the override_si_basis amount.

## Screen Layout

**Header**: Run reference (Pay Group, Period, Run ID), total exception count, open exception count.

**Filter bar**:
- Exception type: multi-select (NEGATIVE_NET / MIN_WAGE_VIOLATION / OT_CAP_EXCEEDED / ZERO_BHXH_SI_ELIGIBLE / ALL)
- Status: OPEN / ACKNOWLEDGED / PENDING_FINANCE_APPROVAL / ALL
- Search: free-text on worker name

**Exception Table**:

| Column | Notes |
|--------|-------|
| Worker Name | |
| Employee ID | |
| Exception Type | Colored badge |
| Detail | One-line summary (e.g., "Net: –450,000 VND") |
| Status | OPEN / ACKNOWLEDGED / PENDING_FINANCE_APPROVAL |
| Acknowledged By | Empty if OPEN |
| Acknowledged At | Empty if OPEN |
| Actions | [Acknowledge] opens inline resolution panel; [View Calc Log] deep link |

Row expansion: clicking a row expands to show full exception detail, the worker's gross / SI / PIT / net amounts, and the resolution form.

## Bulk Acknowledge

**Trigger**: [Bulk Acknowledge] button at top of exception table.

**Flow**:
1. Admin checks multiple rows (same exception type is not required, but the resolution options differ per type).
2. Click [Bulk Acknowledge Selected].
3. System groups selected exceptions by type. For each group, shows the resolution options for that type.
4. Admin chooses resolution option + reason_code for each group.
5. Confirm: applies the acknowledgement to all selected rows.

Limitations:
- SET_TO_ZERO (write-off) cannot be bulk-applied. Each must be individually acknowledged because Finance Manager co-approval is required per worker.
- FORCE_INCLUDE (ZERO_BHXH_SI_ELIGIBLE) cannot be bulk-applied due to the per-worker basis amount input.

---

## Individual Acknowledge — Modal

**Trigger**: [Acknowledge] on a single row.

Modal contents:
1. Exception type badge and description.
2. Worker details: name, employee ID, gross, net, exception detail.
3. Resolution option selector (radio or dropdown, per exception type).
4. Conditional additional inputs (see per-type specs above).
5. Reason note field (free text, 500 chars).
6. [Confirm Acknowledgement] button.

After confirming: row status updates to ACKNOWLEDGED (or PENDING_FINANCE_APPROVAL for SET_TO_ZERO). Toast: "Exception acknowledged for [Worker Name]."

## Export

[Export CSV] button downloads all exceptions (matching current filter) as a CSV file.

CSV columns: run_id, exception_type, worker_name, employee_id, exception_detail, status, resolution_option, reason_code, acknowledged_by, acknowledged_at, note.

File name: `exceptions_{pay_group}_{period}_{timestamp}.csv`

## Business Rules

| Rule | Description |
|------|-------------|
| BR-090 | NEGATIVE_NET exception raised whenever net < 0 |
| BR-091 | MIN_WAGE_VIOLATION raised when gross < minimum wage for worker's region |
| BR-092 | OT_CAP_EXCEEDED raised when OT hours > pay group / profile cap |
| BR-094 | ZERO_BHXH_SI_ELIGIBLE raised when SI-eligible contract_type has SI_BASIS = 0 |
| — | All exceptions must be acknowledged before run submittable for approval |
| — | Acknowledgement is immutable once made; correction requires creating a new run |

---

## Error Handling

| Scenario | Behavior |
|----------|---------|
| Acknowledge attempted after period locked | Action blocked: "This period is locked. Exception records are read-only." |
| Finance Manager rejects SET_TO_ZERO | Exception returns to OPEN status with rejection reason. Admin must choose ZERO_NET_WITH_RECOVERY instead, or correct the source data and re-run. |
| All exceptions acknowledged but run is DRY_RUN | [Submit for Approval] remains disabled; tooltip: "DRY_RUN results cannot be submitted for approval. Initiate a PRODUCTION run." |
