# PR-M-003: Pay Profile Management

**Type**: M (Masterdata)
**Priority**: P0
**BC**: BC-01 — Pay Master Configuration
**Actors**: PAYROLL_ADMIN (primary)
**Linked Stories**: US-003, US-004, US-005
**API Endpoints**:
- `POST /api/v1/payroll-domain/pay-profiles` — Create pay profile
- `GET /api/v1/payroll-domain/pay-profiles` — List pay profiles
- `GET /api/v1/payroll-domain/pay-profiles/{id}` — Get pay profile detail

- `DELETE /api/v1/payroll-domain/pay-profiles/{id}/pay-element-bindings/{binding_id}` — Remove element binding

## Overview

A Pay Profile defines how a worker's gross pay is calculated. It combines a pay method (MONTHLY_SALARY / HOURLY / PIECE_RATE / GRADE_STEP), proration rules, rounding method, and a set of pay element bindings. The profile may inherit base configuration from a parent profile, overriding only specific fields.

The primary UX challenge is the dynamic form: selecting a `pay_method` reveals method-specific configuration sections while hiding irrelevant ones. The admin stays in one form rather than navigating between separate screens.

## Business Rules

- **BR-001**: Proration applies when a worker has less than a full period of work; the `proration_method` on the profile is the default; element-level setting overrides it.
- **BR-005**: HOURLY profiles require an `hourly_rate_config` table mapping shift_type × grade to a rate.
- **BR-006**: PIECE_RATE profiles require a `piece_rate_table` mapping product_code × quality_grade to a rate per unit.
- **BR-007**: GRADE_STEP profiles require a `pay_scale_table` mapping grade × step to a coefficient or a fixed amount.
- A profile code must be unique per legal entity.
- A profile cannot be deleted if it is currently assigned to any worker (active or historical). Deactivation is the only allowed action.
- Inheritance: a profile with a `parent_profile_id` inherits all element bindings and rule bindings from the parent. Child settings override parent settings field-by-field. Circular inheritance is rejected on save.
- OT cap hours overrides the pay group-level `ot_monthly_cap` for workers on this profile.

## CRUD Specification

### Fields — Core (always visible)

| Field | Type | Required | Validation | Notes |
|-------|------|----------|------------|-------|
| profile_code | String | Yes | Unique per legal_entity; alphanumeric + hyphens; max 50 chars | Immutable after first worker assignment |
| profile_name | String | Yes | Max 200 chars | Display name |
| legal_entity_id | UUID | Yes | Must exist in org_legal.entity | Scopes profile to one legal entity |
| pay_method | Enum | Yes | MONTHLY_SALARY / HOURLY / PIECE_RATE / GRADE_STEP | Controls dynamic sections shown below |
| proration_method | Enum | Yes | CALENDAR_DAYS / WORK_DAYS / NONE | Profile-level default |
| rounding_method | Enum | Yes | ROUND_HALF_UP / ROUND_DOWN / TRUNCATE | Applied to final net amount |
| parent_profile_id | UUID | No | Must exist and belong to same legal entity; no circular reference | Enables inheritance |
| ot_cap_hours_month | Integer | No | Min: 1; Max: 200; Default: inherits pay group setting | Profile-specific OT cap |
| description | String | No | Max 500 chars | Free-text notes |
| status | Enum | System | ACTIVE / INACTIVE | Default: ACTIVE on creation |

### Dynamic Sections per pay_method

#### pay_method = MONTHLY_SALARY

| Field | Type | Required | Validation |
|-------|------|----------|------------|
| base_salary_min | Decimal | No | >= 0; currency = profile's legal entity currency |
| base_salary_max | Decimal | No | >= base_salary_min |

Notes: min/max are advisory validation bands shown at worker assignment time. Not enforced as hard constraints.

#### pay_method = HOURLY

Hourly Rate Config table. Each row represents a rate for a combination of `shift_type` and `grade_code`:

| Column | Type | Required | Validation |
|--------|------|----------|------------|
| shift_type | Enum | Yes | REGULAR / OVERTIME / NIGHT / WEEKEND |
| grade_code | String | Yes | Must exist in HR grade table |
| hourly_rate | Decimal | Yes | > 0 |
| effective_from | Date | Yes | Cannot overlap with another row of same shift_type + grade_code |

UI: inline editable table with [Add Row] and [Remove Row] actions. Duplicate shift_type + grade_code combination is rejected on save.

#### pay_method = PIECE_RATE

Piece Rate Table. Each row represents a rate for a combination of `product_code` and `quality_grade`:

| Column | Type | Required | Validation |
|--------|------|----------|------------|
| product_code | String | Yes | Max 50 chars; free text |
| quality_grade | String | Yes | Max 20 chars (e.g., A, B, C, REJECT) |
| rate_per_unit | Decimal | Yes | > 0 |
| unit | String | Yes | Max 20 chars (e.g., PCS, KG, BOX) |
| effective_from | Date | Yes | |

UI: inline editable table with [Add Row] and [Remove Row] actions.

#### pay_method = GRADE_STEP

Pay Scale Table. Each row represents the pay amount or coefficient for a grade × step combination:

| Column | Type | Required | Validation |
|--------|------|----------|------------|
| grade_code | String | Yes | Must exist in HR grade table |
| step_code | String | Yes | Must exist in HR step table for that grade |
| value_type | Enum | Yes | COEFFICIENT / FIXED_AMOUNT |
| value | Decimal | Yes | > 0 |
| effective_from | Date | Yes | |

UI: inline editable table. On `value_type = COEFFICIENT`, a tooltip explains: "Coefficient × VN_LUONG_CO_SO (statutory_rule) = gross salary."

#### pay_method = TASK_BASED

**V1: Disabled.** Section shows a locked banner: "Task-Based pay configuration is planned for V2. This pay method cannot be selected in V1."

### Pay Element Bindings

Tab: **Pay Elements**

Displays a list of pay elements bound to this profile. Bindings are effective-dated.

| Column | Notes |
|--------|-------|
| Element Code | Links to PR-M-002 detail |
| Element Name | |
| Classification | EARNING / DEDUCTION / EMPLOYER_CONTRIBUTION |
| Effective Date | Date this element applies from |
| End Date | Optional; leave blank for open-ended |
| Inherited | Badge if binding comes from parent profile |

Actions:
- [Add Element]: opens a modal search to select a pay element; set effective_date and optional end_date.
- [Remove]: sets an end_date = today (not a hard delete). Inherited bindings cannot be removed on the child profile — must be removed on the parent.

### Rule Bindings

Tab: **Rules**

Two rule types configurable per profile:

| Rule | Options |
|------|---------|
| Minimum wage check | ENFORCE (default) / WARN_ONLY / DISABLED |
| SI basis override | Use Profile Default / Override with specific statutory_rule_id |

## Error Handling

| Scenario | Behavior |
|----------|---------|
| Circular parent reference | Save rejected: "Circular inheritance detected. Profile A → B → A is not allowed." |
| Deleting profile with active workers | Delete blocked; message: "N workers are currently assigned to this profile. Deactivate instead." |
| Duplicate hourly_rate row (same shift_type + grade_code + overlapping dates) | Inline row validation error on table |
| pay_method = TASK_BASED selected | Select option disabled in V1 with tooltip explanation |
| Missing rate table rows for HOURLY/PIECE_RATE/GRADE_STEP | Save allowed with warning: "Rate table is empty. Workers on this profile will produce zero variable pay." |
