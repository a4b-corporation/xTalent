# PR-M-004: Worker Pay Assignment

**Type**: M (Masterdata)
**Priority**: P0
**BC**: BC-01 — Pay Master Configuration
**Actors**: PAYROLL_ADMIN (primary), HR_MANAGER (can initiate, PAYROLL_ADMIN confirms)
**Linked Stories**: US-006, US-007
**API Endpoints**:
- `POST /api/v1/payroll-domain/pay-assignments` — Create assignment
- `GET /api/v1/payroll-domain/pay-assignments` — List assignments (filterable)
- `GET /api/v1/payroll-domain/pay-assignments/{id}` — Get assignment detail
- `PUT /api/v1/payroll-domain/pay-assignments/{id}` — Update assignment (future-dated only)
- `POST /api/v1/payroll-domain/pay-assignments/{id}/terminate` — End assignment
- `POST /api/v1/payroll-domain/pay-assignments/bulk-import` — Bulk import via CSV

---

## Overview

A Worker Pay Assignment links a working relationship (from Core Organization module) to a Pay Group and a Pay Profile. It is effective-dated so that changes in pay structure create a new version of the assignment rather than modifying the current one. The assignment record also captures grade/step codes for workers on GRADE_STEP profiles.

## Business Rules

- **BR-100**: The `legal_entity_id` of the working relationship must match the `legal_entity_id` of the Pay Group. Mismatched assignments are rejected.
- **BR-101**: A worker (working_relationship_id) cannot be enrolled in two active Pay Groups simultaneously. Enrolling a worker who is already active in another Pay Group requires terminating the prior assignment first.
- Assignment history is immutable once a period that used that assignment has been locked. Future-dated assignments can be updated.
- Effective date must be >= today (past-dating requires PAYROLL_ADMIN with explicit justification reason stored as a note).
- Grade and step codes are required when the selected pay profile has `pay_method = GRADE_STEP`; optional otherwise.
- Bulk import via CSV follows the same validation rules as individual assignment. Rows with errors are listed in an error report; valid rows are imported.

## CRUD Specification

### Fields — Individual Assignment

| Field | Type | Required | Validation | Notes |
|-------|------|----------|------------|-------|
| working_relationship_id | UUID | Yes | Must exist in CO module; fetch name and legal_entity for display | Lookup from CO |
| pay_group_id | UUID | Yes | Must belong to same legal_entity as the working_relationship | Select filtered by legal entity |
| pay_profile_id | UUID | Yes | Must belong to same legal_entity; must be ACTIVE | Select filtered by legal entity |
| effective_date | Date | Yes | >= today (past-dating allowed with reason) | Start of this assignment version |
| end_date | Date | No | Must be > effective_date if provided | Leave blank for open-ended |
| grade_code | String | Conditional | Required when profile pay_method = GRADE_STEP | Must exist in HR grade table |
| step_code | String | Conditional | Required when profile pay_method = GRADE_STEP | Must exist within grade_code's steps |
| assignment_note | String | No | Max 500 chars; required for past-dated effective_date | Reason for back-dating |

### Lookup: Worker Search

The `working_relationship_id` field uses a typeahead search panel:
- Search by: worker name, employee ID, or working_relationship_id
- Results show: worker name, employee ID, department, legal entity, current pay group (if any)
- Workers already actively assigned to a pay group in the same legal entity show a warning badge: "Already assigned to [Pay Group Name]"

### Cascade on Pay Profile Selection

After selecting `pay_profile_id`:
- If profile `pay_method = GRADE_STEP`: grade_code and step_code fields appear; they are required.
- If profile `pay_method != GRADE_STEP`: grade_code and step_code fields are hidden.
- System displays a read-only preview of the profile's proration_method and active element bindings count.

## Bulk Assignment — CSV Import

Access: [Bulk Import] button on the assignment list screen.

### CSV Format

Required columns (order-independent, header row mandatory):

| Column | Format | Required |
|--------|--------|---------|
| working_relationship_id | UUID | Yes |
| pay_group_code | String | Yes |
| pay_profile_code | String | Yes |
| effective_date | YYYY-MM-DD | Yes |
| grade_code | String | Conditional (required for GRADE_STEP profiles) |
| step_code | String | Conditional (required for GRADE_STEP profiles) |
| assignment_note | String | No |

### Import Workflow

1. **Upload**: drag-and-drop or file picker (.csv, max 5 MB, max 1000 rows).
2. **Validation preview**: system validates all rows before committing. Shows:
   - Total rows detected
   - Valid rows count
   - Error rows count with error detail per row (row number, field, error message)
3. **Partial import**: admin can choose "Import valid rows only" (skip error rows) or "Cancel and fix all errors first."
4. **Confirm**: confirmation modal shows the count of workers to be assigned and the effective date.
5. **Result**: success toast with count. Error rows downloadable as a CSV report.

### Download Template

[Download CSV Template] link on the import screen. Includes a header row and one example row with comments.

## Error Handling

| Scenario | Behavior |
|----------|---------|
| Worker already in active pay group | Blocking warning in lookup results; save rejected if proceeding without terminating prior assignment |
| Legal entity mismatch | Inline error: "Pay Group [X] belongs to [Legal Entity B]. Worker's legal entity is [Legal Entity A]. They must match." |
| Grade/step required but not provided | Inline required field error: "Grade and Step are required for GRADE_STEP profiles." |
| CSV row with unknown pay_group_code | Per-row error in validation preview: "Row 7: pay_group_code 'SG-ALL' not found in legal entity [X]." |
| Past-dated effective_date without note | Save blocked: "Back-dated assignments require an assignment note explaining the reason." |
