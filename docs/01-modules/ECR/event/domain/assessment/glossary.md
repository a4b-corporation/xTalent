# Glossary: Assessment
## ECR Domain | 2026-03-25

This glossary defines the ubiquitous language for the Assessment bounded context. **This context is an interface boundary only.** Internal assessment domain models are intentionally undefined pending the build vs. integrate decision. All terms here describe the contract surface, not internal mechanics.

---

| Term | Definition | Notes / Disambiguations |
|------|-----------|------------------------|
| **Assessment** | In the ECR domain: the act of evaluating a candidate's knowledge or skills using a structured test. Assessment is delivered to eligible candidates during an Event. | The internal implementation (test engine, question bank, scoring algorithm) is out of scope for ECR domain design. ECR only models the interface contract: dispatch and result ingestion. |
| **Blueprint** | In the Assessment context: the configuration template specifying which questions, time limits, scoring criteria, and passing thresholds apply to an assessment delivered for a specific Track. | Disambiguate from "Event Blueprint" (Event Management clone template). A Blueprint is an assessment template. Each Track in an Event is associated with a BlueprintID. Whether a Blueprint is instantiated per-delivery or shared is an internal detail of the Assessment Engine, not exposed to ECR. |
| **BlueprintID** | An opaque identifier referencing a specific assessment Blueprint in the external Assessment Engine. ECR treats this as a foreign key — its internal structure is unknown and irrelevant to the ECR domain model. | Not a UUID generated by ECR. BlueprintID is assigned and managed by the Assessment Engine. ECR stores it as a configuration value on each Track. |
| **AssessmentResult** | The output produced by the Assessment Engine for a single candidate's completed assessment. Ingested by the ECR ACL and translated to an `AssessmentCompleted` domain event. Contains: candidateRef, blueprintId, completedAt, score, passed, rawData. | `rawData` is opaque to ECR — it is passed through unmodified and stored for audit purposes. ECR domain logic uses only `score` and `passed`. |
| **Delivery Gate** | The precondition that restricts assessment dispatch to candidates with `registration_status = CONFIRMED` or `check_in_status = CHECKED_IN`. Enforced by the ACL before calling the Assessment Engine (BR-07). | "Delivery Gate" is not a feature flag — it is a business rule (BR-07) implemented as a mandatory guard in the ACL. It cannot be bypassed by any actor including TA Coordinators. |
| **ACL (Anti-Corruption Layer)** | The translation layer between the ECR domain and the external Assessment Engine. Responsible for: converting ECR candidate references to the engine's format, enforcing the delivery gate (BR-07), ingesting results and translating to ECR domain events, shielding ECR from engine API changes. | The ACL is an infrastructure concern, not a domain class. It is documented here because it is the only visible interface of the Assessment bounded context to the rest of ECR. |
| **CandidateAssessmentRef** | A value object used as the input to the Assessment Engine. Composed of: `eventId`, `sbdNumber`, `trackId`, `blueprintId`. Identifies the candidate and assessment configuration without exposing internal registration IDs to the external engine. | This is the "translation key" the ACL uses. The external engine receives this reference and returns results using the same reference for correlation. |
| **AssessmentDispatch** | The act of sending an assessment link or access token to an eligible candidate. Triggers `AssessmentDispatched` domain event. Must be preceded by a successful Delivery Gate check. | Assessment dispatch is not the same as Communication dispatch. The Assessment Engine may handle delivery directly, or ECR's Communication BC may be involved. The interface contract is agnostic on this point. |
| **Assessment Eligibility** | A candidate is eligible for assessment if their `registration_status = CONFIRMED` OR their `check_in_status = CHECKED_IN` (BR-07). Eligibility is evaluated at dispatch time, not pre-computed. | If a candidate's status changes between confirmation and dispatch (e.g., cancelled after confirmation), the ACL re-checks eligibility at dispatch time. |

---

## Interface Contract (Formal)

```
DISPATCH INPUT:
  CandidateAssessmentRef:
    eventId:      string (ECR event UUID)
    sbdNumber:    string (event-scoped SBD)
    trackId:      string (ECR track UUID)
    blueprintId:  string (opaque ID from Assessment Engine)
    deliveryContext:
      dispatchedAt:  datetime
      dispatchedBy:  string (actorId or "system")

RESULT OUTPUT (from Assessment Engine webhook/callback):
  AssessmentResult:
    candidateRef:  CandidateAssessmentRef (echo of input)
    completedAt:   datetime
    score:         decimal (0.0 – 100.0)
    passed:        boolean
    rawData:       JSON (opaque — stored as-is)
```

## Business Rules in This Context

| BR-ID | Rule | ACL Implementation |
|-------|------|-------------------|
| BR-07 | Assessment delivery only to Confirmed/Checked-In candidates | ACL checks registration status before calling Assessment Engine. Rejects dispatch if status is not eligible. |
