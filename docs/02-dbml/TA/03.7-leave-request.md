# 3.7 · LeaveRequest — Workflow nghỉ phép

> **Bounded context:** Absence · **Layer:** Operational (Workflow)  
> **DBML table:** `absence.leave_request`  
> **Upstream:** `absence.leave_instant`, `absence.holiday_calendar`, `employment.employee`  
> **Downstream:** `absence.leave_reservation`, `absence.leave_movement`, Temporal Workflow Engine

---

## Mục đích nghiệp vụ

`LeaveRequest` là **đơn xin nghỉ phép** của nhân viên. Entity này:
- Theo dõi vòng đời request (DRAFT → ... → APPROVED/CANCELLED)
- Kết nối với balance (LeaveInstant) qua mechanism RESERVE → USE
- Tích hợp với Temporal Workflow Engine để xử lý multi-level approval
- Ghi lại lịch sử phê duyệt (approved_by, approved_at, rejection_reason)

---

## Entity Schema

### 1. Identity

| Field | Type | Nullable | Mô tả |
|-------|------|----------|-------|
| `id` | UUID | NOT NULL | PK |
| `employee_id` | UUID (FK) | NOT NULL | FK → `employment.employee.id` |
| `class_id` | UUID (FK) | NOT NULL | FK → `absence.leave_class.id` — loại phép nào |
| `instant_id` | UUID (FK) | NULLABLE | FK → `absence.leave_instant.id` — balance nào bị ảnh hưởng |

### 2. Leave Dates

| Field | Type | Nullable | Mô tả |
|-------|------|----------|-------|
| `start_dt` | timestamp | NOT NULL | Ngày giờ bắt đầu nghỉ |
| `end_dt` | timestamp | NOT NULL | Ngày giờ kết thúc nghỉ |
| `total_days` | decimal(6,2) | NULLABLE | Số ngày nghỉ thực tế (working days, sau khi exclude holidays). Computed khi submit |
| `qty_hours_req` | decimal(6,2) | NULLABLE | Số giờ nghỉ (dùng khi `unit_code=HOUR`) |
| `is_half_day` | boolean | default: false | Request half-day |
| `half_day_period` | varchar(10) | NULLABLE | `MORNING` hoặc `AFTERNOON`. Chỉ valid khi `is_half_day=true` và `leave_type.allows_half_day=true` |

### 3. Calendar Reference — Audit-ready

| Field | Type | Nullable | Mô tả |
|-------|------|----------|-------|
| `holiday_calendar_id` | UUID (FK) | NULLABLE | FK → `absence.holiday_calendar.id`. **Frozen tại submit time** — ghi nhận calendar nào được dùng để compute `total_days`. Đảm bảo retro audit không cần re-run calendar logic |

**Tại sao freeze calendar tại submit?**
```
Scenario: Holiday calendar được update sau khi request APPROVED:
  - Nếu không freeze: re-compute total_days → số ngày nghỉ thay đổi → audit fail
  - Nếu freeze: total_days cố định theo calendar tại thời điểm submit → consistent
```

### 4. Status Lifecycle (Unified Standard — Change 40)

| Field | Type | Nullable | Mô tả |
|-------|------|----------|-------|
| `status_code` | varchar(20) | default: 'DRAFT' | Xem state machine bên dưới |

#### Status Machine — Đầy đủ

```
                       ┌─────────────────────────────────────────────────┐
                       │                                                  │
        ┌──────────┐   │  ┌───────────┐    ┌────────────┐                │
  NV →  │  DRAFT   ├───┴→─┤ SUBMITTED ├──→─┤UNDER_REVIEW├─→─ APPROVED → │
        └──────────┘      └─────┬─────┘    └─────┬──────┘       │       │
                                │                 │               │       │
                                │ self-withdraw   │ any reject    │ [leave starts]
                                ↓                 ↓               ↓       │
                           WITHDRAWN           REJECTED          USE      │
                                                                movement   │
                                                                          │
        APPROVED ──────────────────────────── after cancellation deadline │
                │                                                         │
                │ cancel (before deadline) ────────────────────────────→ CANCELLED
                │                                                         │
                │ cancel (after deadline)──→ CANCELLATION_PENDING ──────→ (manager decides)
                │                                 │                       │
                │                                 ├──→ CANCELLED ←───────┘
                │                                 └──→ (stay APPROVED, cancel rejected)
```

#### Trạng thái và ý nghĩa

| Status | Ai tạo | Balance effect | Mô tả |
|--------|--------|---------------|-------|
| `DRAFT` | Employee | Không có | Lưu nháp, chưa submit |
| `SUBMITTED` | Employee (action: submit) | **RESERVE −** | Đã submit. Reservation tạo. Routing bắt đầu |
| `UNDER_REVIEW` | Temporal Workflow Engine | Hold duy trì | Ít nhất 1 approver đang review |
| `APPROVED` | Approver (cuối cùng) | Hold duy trì → **USE −** khi bắt đầu | Tất cả approved |
| `REJECTED` | Bất kỳ approver nào | **RELEASE +** | Bị từ chối |
| `CANCELLATION_PENDING` | Employee (sau deadline) | Hold duy trì | Cancel sau deadline, cần manager chấp thuận |
| `CANCELLED` | Employee (trước deadline) hoặc Manager (từ CANCELLATION_PENDING) | **RELEASE +** | Đã cancel |
| `WITHDRAWN` | Employee (trước UNDER_REVIEW) | **RELEASE +** | Employee tự rút trước khi ai xem |

#### Deadline logic for Cancel vs Withdraw

```
cancellation_deadline_days = leave_type.cancellation_deadline_days (default = 0)

IF request.start_dt - today > cancellation_deadline_days:
    Self-cancel → CANCELLED (no manager needed)
ELSE:
    Cancel → CANCELLATION_PENDING (requires manager approval)

WITHDRAW (before anyone acts):
    status == SUBMITTED and no approver has acted yet
    → Employee withdraws → WITHDRAWN
```

## `total_days` Calculation Logic

```python
# Pseudo-code: compute total_days when DRAFT → SUBMITTED
def compute_total_days(request):
    leave_type = get_leave_type(request.class_id)
    calendar_id = resolve_holiday_calendar(request.employee_id)

    if leave_type.holiday_handling == 'EXCLUDE':
        total_days = count_working_days(
            start=request.start_dt.date(),
            end=request.end_dt.date(),
            calendar_id=calendar_id
        )
    else:  # INCLUDE
        total_days = (request.end_dt.date() - request.start_dt.date()).days + 1

    if request.is_half_day:
        total_days = 0.5

    request.total_days = round(total_days, 1)
    request.holiday_calendar_id = calendar_id  # freeze ngay tại đây
    return total_days
```

**Resolve `holiday_calendar_id`:**
```
employee_id → employment.employee.org_unit_id → BU → country_code 
→ SELECT holiday_calendar WHERE country_code = 'VN' AND is_active = true
→ Nếu nhiều calendar: ưu tiên calendar có scope nhỏ hơn (BU-level > Country-level)
```

## Validation Rules tại Submit

Tất cả check này chạy khi DRAFT → SUBMITTED:

```
1. Balance check:
   leave_instant.available_qty >= request.total_days
   (với leave_type.allows_negative=false)

2. Blackout period check:
   Không có leave_blackout_period active overlapping request dates
   (scope: company-wide hoặc employee's org_unit)

3. LIMIT policy check:
   used_ytd + total_days <= limit_yearly
   total_days <= max_per_request
   total_days >= min_duration

4. Advance notice check:
   (start_dt - today).days >= leave_type.min_advance_days

5. Half-day validation:
   is_half_day=true → leave_type.allows_half_day=true
   half_day_period IN ('MORNING', 'AFTERNOON')

6. Certificate check (async, post-submit):
   leave_type rules_json.requires_certificate_after_days → reminder job

7. Team limit check:
   team_leave_limit.limit_pct/limit_abs_cnt kiểm tra concurrent absences trong org_unit
```

## Integration Points

```
absence.leave_request
  ← employment.employee (requestor)
  ← absence.leave_class (what type of leave)
  ← absence.leave_instant (whose balance)
  ← absence.holiday_calendar (which calendar for total_days, frozen at submit)
  → absence.leave_reservation (1:1 hold mechanism)
  → absence.leave_movement (RESERVE, USE, RELEASE events)
  ← person.document (certificate_doc_id, soft ref)
  ↔ Temporal Workflow Engine (approval lifecycle)
```

## Common Gotchas

1. **WITHDRAWN vs CANCELLED:** `WITHDRAWN` = employee tự rút trước khi bất kỳ approver nào có action. `CANCELLED` = cancel sau khi đã được xem xét hoặc approved. Temporal engine phân biệt 2 transition này.

2. **`holiday_calendar_id` phải freeze tại submit, không phải tại approve:** Timestamp quan trọng là khi request đi vào hệ thống, không phải khi được duyệt.

3. **Sick leave certificate là async:** Employee submit trước, nộp giấy sau (trong 48h). Hệ thống phải có reminder job check `certificate_doc_id IS NULL` cho sick leave requests > N ngày.

4. **Multi-level approval detail không lưu ở đây:** Temporal Workflow Engine quản lý approval chain. `leave_request` chỉ lưu kết quả cuối (approved_by, approved_at). Nếu cần xem audit trail từng level → query Temporal history.

5. **`total_days` computation phải consistent:** Nếu leave type `holiday_handling=EXCLUDE` và request gắn với calendar VN-2026 có 1/5/2026 là lễ Quốc tế Lao động — request từ 30/4 đến 2/5 = 2 working days (không phải 3). `holiday_calendar_id` đảm bảo computation này có thể verify lại.
