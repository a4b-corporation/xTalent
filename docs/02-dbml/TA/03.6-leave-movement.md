# 3.6 · LeaveMovement — Sổ cái bất biến (Immutable Ledger)

> **Bounded context:** Absence · **Layer:** Operational (Ledger)  
> **DBML table:** `absence.leave_movement`  
> **Upstream:** `absence.leave_instant`, `absence.leave_event_run`, `absence.leave_request`  
> **Downstream:** `absence.leave_instant` (balance recomputed), Payroll export

---

## Mục đích nghiệp vụ

`LeaveMovement` là **sổ cái bất biến** (immutable ledger) ghi lại **mọi thay đổi** trong balance nghỉ phép. Không có record nào trong bảng này được phép update hay delete — mọi điều chỉnh đều là một record mới.

**Nguyên tắc thiết kế (ADR-TA-001):**
- **Append-only:** Chỉ INSERT, không UPDATE, không DELETE
- **Idempotent:** `idempotency_key` ngăn duplicate posting khi retry
- **Source of truth:** Toàn bộ balance của `LeaveInstant` có thể được tính lại bằng cách `SUM(qty)` từ ledger
- **Audit-ready:** Mọi movement đều có link đến event_run hoặc request

## Entity Schema

### 1. Identity & Routing

| Field | Type | Nullable | Mô tả |
|-------|------|----------|-------|
| `id` | UUID | NOT NULL | PK |
| `instant_id` | UUID (FK) | NOT NULL | FK → `absence.leave_instant.id` — batch này affect balance nào |
| `class_id` | UUID (FK) | NOT NULL | FK → `absence.leave_class.id` — denorm để query audit per class |

### 2. Event & Quantity

| Field | Type | Nullable | Mô tả |
|-------|------|----------|-------|
| `event_code` | varchar(50) | NOT NULL | Loại movement — xem bảng Event Codes bên dưới |
| `qty` | decimal(10,2) | NOT NULL | **Positive = tăng balance. Negative = giảm balance** |
| `unit_code` | varchar(10) | NOT NULL | `DAY` hoặc `HOUR` — phải match với `leave_class.unit_code` |

### 3. Timing & Period

| Field | Type | Nullable | Mô tả |
|-------|------|----------|-------|
| `effective_date` | date | NOT NULL | Ngày movement có hiệu lực (không nhất thiết là ngày post) |
| `expire_date` | date | NULLABLE | Ngày hết hạn nếu movement này cấp balance có expiry (dùng cho EARN/CARRY) |
| `posted_at` | timestamptz | default: now() | Ngày thực tế record được insert vào DB |
| `period_id` | UUID (FK, null) | NULLABLE | FK → `absence.leave_period.id`. Liên kết kỳ kế toán (chỉ áp dụng cho batch events) |

### 4. Traceability — Nguồn gốc movement

| Field | Type | Nullable | Mô tả |
|-------|------|----------|-------|
| `request_id` | UUID | NULLABLE | FK → `absence.leave_request.id`. Không null nếu movement do leave request tạo ra (RESERVE, USE, RELEASE) |
| `lot_id` | UUID | NULLABLE | FK → `absence.leave_instant_detail.id`. Không null nếu movement link đến một lot cụ thể (FEFO traceability) |
| `run_id` | UUID | NULLABLE | Soft reference → `absence.leave_event_run.id`. Không null nếu movement do batch event run tạo ra (EARN, CARRY, EXPIRE) |

> **`run_id` là soft reference (không FK cứng):** Movement có thể được tạo bởi:
> - Batch event run → `run_id` = event_run.id, `request_id` = null
> - Leave request action → `request_id` = request.id, `run_id` = null
> - Manual HR adjustment → cả hai đều null (dùng `metadata.reason` để trace)
>
> Đây là thiết kế intentional — tránh circular dependency và cho phép manual movements.

### 5. Idempotency

| Field | Type | Nullable | Mô tả |
|-------|------|----------|-------|
| `idempotency_key` | varchar(120) | NULLABLE, UNIQUE | Ngăn tạo duplicate movement khi retry. Format: `{event_code}-{employee_code}-{period}-{suffix}` |

**Format idempotency_key:**
```
EARN-EMP001-FY2026-JAN          → Monthly accrual January 2026
CARRY-EMP001-FY2025-TO-2026     → Year-end carry-over
RESERVE-REQ-a1b2c3d4            → Reserve for specific request (short UUID)
USE-REQ-a1b2c3d4-2026-03-15     → Use for specific request at start date
```

### 6. Multi-entity Scope

| Field | Type | Nullable | Mô tả |
|-------|------|----------|-------|
| `bu_id` | UUID | NULLABLE | Business Unit — cho phép filter/audit by BU |
| `le_id` | UUID | NULLABLE | Legal Entity — cho phép audit by LE |
| `country_code` | varchar(10) | NULLABLE | Country — phục vụ multi-country payroll export |

### 7. Metadata

| Field | Type | Nullable | Mô tả |
|-------|------|----------|-------|
| `metadata` | jsonb | NULLABLE | Thông tin bổ sung: lý do adjust, approver, notes |

## Event Codes — Ý nghĩa từng loại movement

| `event_code` | qty sign | Mô tả | Source |
|-------------|---------|-------|--------|
| `EARN` / `ACCRUAL` | **+** | Cấp balance mới (grant, accrual) | Event run (ACCRUAL type) |
| `CARRY` | **+** | Cộng carry-over từ kỳ trước vào kỳ mới | Event run (CARRY type) |
| `RESERVE` | **−** | Tạm hold balance khi request SUBMITTED | Leave request (SUBMITTED state) |
| `USE` | **−** | Trừ balance thực sự khi leave bắt đầu | Leave request (APPROVED + leave start date) |
| `RELEASE` | **+** | Trả lại balance từ hold khi request REJECTED/CANCELLED/WITHDRAWN | Leave request status change |
| `ADJUST` | **+ hoặc −** | Điều chỉnh manual bởi HR | Manual HR action |
| `EXPIRE` | **−** | Trừ balance hết hạn | Event run (EXPIRE type) |
| `CASHOUT` | **−** | Trừ balance khi đổi thành tiền | Encashment workflow (approved) |

## Movement Lifecycle — Full Sequence

```
[1] Employee submit request:
    → INSERT leave_movement(event_code=RESERVE, qty=-3.0, request_id=req-001)
    → leave_instant.hold_qty += 3.0, available_qty -= 3.0

[2a] Request APPROVED (leave start date arrives):
    → INSERT leave_movement(event_code=USE, qty=-3.0, request_id=req-001)
    → leave_instant.current_qty -= 3.0, hold_qty -= 3.0 (available_qty unchanged)
    → leave_instant.used_ytd += 3.0

[2b] Request REJECTED/WITHDRAWN/CANCELLED:
    → INSERT leave_movement(event_code=RELEASE, qty=+3.0, request_id=req-001)
    → leave_instant.hold_qty -= 3.0, available_qty += 3.0
    Note: current_qty KHÔNG thay đổi (RESERVE không giảm current)
```

**Balance equation luôn đúng:**
```
available_qty  = SUM(qty WHERE event_code IN (EARN, CARRY, RELEASE, ADJUST+, CASHOUT, USE, EXPIRE, ADJUST-))
hold_qty       = SUM(-qty WHERE event_code = RESERVE) - SUM(qty WHERE event_code = RELEASE) - used from reservation
current_qty    = available_qty + hold_qty
```

## Idempotency — Retry Safety

```python
# Khi event engine post movement:
try:
    INSERT INTO absence.leave_movement (..., idempotency_key='EARN-EMP001-FY2026')
except UniqueViolation:
    # Movement đã tồn tại — retry an toàn, không double-post
    log.info("Movement already posted, skipping (idempotent)")
    return existing_movement_id
```

## Common Gotchas

1. **KHÔNG UPDATE/DELETE bất kỳ record nào:** Nếu có sai sót, post movement ADJUST (qty = +/-difference) với reason trong metadata.

2. **RESERVE và USE là 2 movements riêng biệt:** Nhiều dev muốn gộp thành 1. Không được — RESERVE là sự kiện "xếp chỗ" (claim), USE là sự kiện "thực sự dùng" (consume). Cho phép audit trail đầy đủ về khoảng thời gian hold.

3. **`run_id` là soft reference (không FK):** `run_id` có thể null (manual movements). Nếu cần join với `leave_event_run`, phải handle null case.

4. **qty sign convention:** `RESERVE` và `USE` phải là **âm** (`-3.0`). `RELEASE` phải là **dương** (`+3.0`). Sai sign = balance bị corrupt. Hệ thống nên validate sign per event_code khi insert.

5. **Recomputing balance từ ledger:**
   ```sql
   SELECT SUM(qty) FROM absence.leave_movement WHERE instant_id = 'xxx'
   -- Phải = leave_instant.current_qty + hold_qty
   -- Nếu không khớp → consistency check failed
   ```
