# 3.8 · LeaveReservation + LeaveReservationLine — Balance Hold (FEFO)

> **Bounded context:** Absence · **Layer:** Operational (Balance Hold)  
> **DBML tables:** `absence.leave_reservation`, `absence.leave_reservation_line`  
> **Upstream:** `absence.leave_request`, `absence.leave_instant`, `absence.leave_instant_detail`  
> **Downstream:** `absence.leave_movement` (USE/RELEASE)

---

## Mục đích nghiệp vụ

`LeaveReservation` là **cơ chế giữ chỗ** (balance hold) ngăn employee dùng cùng ngày nghỉ cho 2 requests khác nhau. Nó được tạo ngay khi request chuyển sang `SUBMITTED`, và bị xoá/released khi request chuyển sang `APPROVED` (converted to USE), `REJECTED`, `CANCELLED`, hoặc `WITHDRAWN`.

`LeaveReservationLine` là chi tiết từng lot FEFO được hold — cho phép hệ thống biết chính xác **lot nào** đang bị hold để khi release, trả đúng về đúng lot.

---

## 3.8.1 LeaveReservation Entity

### Schema

| Field | Type | Nullable | Mô tả |
|-------|------|----------|-------|
| `request_id` | UUID (FK, PK) | NOT NULL | **PK = FK** → `absence.leave_request.id`. 1:1 mapping với leave_request |
| `instant_id` | UUID (FK) | NOT NULL | FK → `absence.leave_instant.id`. Balance nào đang bị hold |
| `reserved_qty` | decimal(10,2) | NOT NULL | Tổng số ngày/giờ đang được hold |
| `expires_at` | timestamptz | NULLABLE | Nếu reservation không được action trước thời điểm này → tự động RELEASE. null = không tự expire |
| `created_at` | timestamptz | default: now() | |

**1:1 với LeaveRequest:**
```
leave_request.id (PK) ←──── leave_reservation.request_id (PK, FK)
```
Vì `request_id` là PK của reservation, mỗi request chỉ có **đúng 1 reservation**. Khi request kết thúc (APPROVED/REJECTED/CANCELLED/WITHDRAWN), reservation record này được DELETE hoặc đánh dấu released (tùy implementation).

### Reservation Lifecycle

```
[DRAFT]      → Không có reservation
[SUBMITTED]  → INSERT leave_reservation + CREATE leave_movement(RESERVE, qty=-N)
              → leave_instant.hold_qty += N
[UNDER_REVIEW] → Reservation maintained (không thay đổi)
[APPROVED]   → Reservation maintained until leave_start_dt
              → On leave start: CREATE leave_movement(USE, qty=-N)
              → Reservation can now be deleted
[REJECTED]   → CREATE leave_movement(RELEASE, qty=+N)
              → DELETE leave_reservation
              → leave_instant.hold_qty -= N
[CANCELLED]  → CREATE leave_movement(RELEASE, qty=+N)
              → DELETE leave_reservation
[WITHDRAWN]  → CREATE leave_movement(RELEASE, qty=+N)
              → DELETE leave_reservation
[CANCELLATION_PENDING] → Reservation maintained
```

### `expires_at` — Tự động expire reservation

Scenario: Employee submit request nhưng sau nhiều ngày không có action từ hệ thống (e.g., approval flow bị stuck). `expires_at` cho phép auto-release sau N ngày:

```
expires_at = submitted_at + max_pending_days (từ leave_policy.VALIDATION.config_json)
```

Scheduled job chạy hàng giờ: `WHERE expires_at < now() AND reservation still active → RELEASE`.

## 3.8.2 LeaveReservationLine Entity

> **Status:** ⚠️ **Thiết kế đã sẵn sàng, dev chưa implement.** Hiện tại dev link `leave_request → leave_reservation` trực tiếp mà không trace lots. `LeaveReservationLine` sẽ được activate khi FEFO engine đầy đủ được implement.

### Schema

| Field | Type | Nullable | Mô tả |
|-------|------|----------|-------|
| `id` | UUID | NOT NULL | PK |
| `reservation_id` | UUID (FK) | NOT NULL | FK → `absence.leave_reservation.request_id` |
| `source_lot_id` | UUID (FK) | NOT NULL | FK → `absence.leave_instant_detail.id` — lot nào đang bị hold |
| `reserved_amount` | decimal(10,2) | NOT NULL | Số ngày hold từ lot cụ thể này |
| `expiry_date` | date | NULLABLE | Copy từ `leave_instant_detail.expire_date` — optimize query "sắp hết hạn" |
| `created_at` | timestamptz | default: now() | |

### Tại sao cần ReservationLine?

Khi FEFO engine chọn lots để hold, cần biết:
1. Hold bao nhiêu từ lot A?
2. Hold bao nhiêu từ lot B?

```
Request: hold 4 ngày
  Lot A (CARRY, expire 31/3): available 3 ngày → hold 3 ngày → ReservationLine 1
  Lot B (GRANT, expire 31/12): available 12 ngày → hold 1 ngày → ReservationLine 2
```

Khi RELEASE (rejected/cancelled):
```
For each ReservationLine:
  lot.used_qty -= line.reserved_amount
→ Balance restored per lot (not bulk)
```

Cho phép accurate release về đúng lot, không mix-up.

## Full FEFO Flow — End to End

```
[1] Employee submit request (4 days, 2026-02-15 to 2026-02-20)

[2] System picks holiday calendar → VN-2026
    Computes total_days:
      2026-02-15 (Mon) ✓, 16 (Tue) ✓, 17 (Wed) ✓, 18 (Thu) ✓, 19 (Fri) ✓
      Wait: 16/2 check against calendar... no holiday
      total_days = 4 (assuming weekend falls on Sat/Sun)

[3] FEFO lot selection:
    Lots available for employee × ANNUAL_STANDARD:
      Lot A: CARRY 3d, expire 2026-03-31, priority=5
      Lot B: GRANT 12d, expire 2026-12-31, priority=10
    Selection:
      Take from Lot A: 3d → still need 1d
      Take from Lot B: 1d → satisfied

[4] INSERT leave_reservation(request_id, instant_id, reserved_qty=4)
    INSERT leave_reservation_line(reservation_id, source_lot_id=A, reserved_amount=3, expiry_date=2026-03-31)
    INSERT leave_reservation_line(reservation_id, source_lot_id=B, reserved_amount=1, expiry_date=2026-12-31)
    INSERT leave_movement(RESERVE, qty=-4, request_id=req-001)

[5] leave_instant.hold_qty += 4 → available_qty -= 4

[6a] Request APPROVED → 2026-02-15 arrives:
     INSERT leave_movement(USE, qty=-4, request_id=req-001)
     leave_instant.current_qty -= 4, hold_qty -= 4
     leave_instant.used_ytd += 4
     UPDATE lot_A.used_qty += 3 (confirmed used)
     UPDATE lot_B.used_qty += 1 (confirmed used)
     DELETE leave_reservation (and lines)

[6b] Request REJECTED:
     INSERT leave_movement(RELEASE, qty=+4, request_id=req-001)
     UPDATE lot_A.used_qty -= 3 (returned)
     UPDATE lot_B.used_qty -= 1 (returned)
     leave_instant.hold_qty -= 4, available_qty += 4
     DELETE leave_reservation (and lines)
```

## Integration Points

```
absence.leave_reservation
  ← absence.leave_request.id (1:1)
  ← absence.leave_instant.id (balance being held)
  → absence.leave_reservation_line (FEFO lot breakdown)
  → absence.leave_movement (RESERVE on create, RELEASE on delete)

absence.leave_reservation_line
  ← absence.leave_reservation.request_id
  ← absence.leave_instant_detail.id (specific lot)
```

## Common Gotchas

1. **`request_id` là PK của reservation:** Không thể có 2 reservations cho cùng 1 request. Đây là intentional 1:1 constraint.

2. **Delete vs soft-delete reservation:** Khi RELEASED — nên DELETE reservation record (và lines). Lịch sử balance đã được ghi vào `leave_movement` (immutable ledger) nên không cần giữ reservation.

3. **Dev hiện tại chưa dùng `LeaveReservationLine`:** Hiện tại chỉ dùng `leave_reservation.reserved_qty` tổng cộng, không trace per-lot. Khi upgrade lên FEFO đầy đủ: phải migrate existing reservations để có lines, hoặc chỉ apply FEFO cho new reservations.

4. **`expires_at` timezone:** Phải lưu theo UTC. Display cho user theo timezone của employee/org_unit. Scheduler cũng loop theo UTC.

5. **Concurrent request race condition:** 2 employees không thể race nhau — tuy nhiên 1 employee có thể submit 2 requests đồng thời (từ 2 browser tabs). DBML unique constraint `(employee_id, class_id)` trên `leave_instant` không bảo vệ khỏi này. **Phải dùng SELECT FOR UPDATE** trên `leave_instant` khi compute available_qty và tạo reservation để tránh double-booking.
