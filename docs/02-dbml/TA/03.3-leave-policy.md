# 3.3 · LeavePolicy + ClassPolicyAssignment — Engine quy tắc nghỉ phép

> **Bounded context:** Absence · **Layer:** Master Data (Configuration)  
> **DBML tables:** `absence.leave_policy`, `absence.class_policy_assignment`  
> **Upstream:** `absence.leave_class`  
> **Downstream:** `absence.leave_event_def`, `absence.leave_event_run`, `absence.leave_instant`

---

## Mục đích nghiệp vụ

`LeavePolicy` là **kho chứa business rules** cho hệ thống nghỉ phép. Thay vì hard-code rules vào từng LeaveClass, policies được định nghĩa độc lập và gắn vào class thông qua `ClassPolicyAssignment` — cho phép tái sử dụng và override linh hoạt.

**Nguyên tắc thiết kế:**
```
1 LeaveClass → nhiều ClassPolicyAssignment → nhiều LeavePolicy (mỗi loại 1)
Ví dụ:
  ANNUAL_STANDARD → ACCRUAL_MONTHLY  (cấp 1 ngày/tháng)
  ANNUAL_STANDARD → CARRY_5DAYS      (carry-over tối đa 5 ngày)
  ANNUAL_STANDARD → LIMIT_STANDARD   (max 20 ngày/năm)
  ANNUAL_STANDARD → SENIORITY_VLC113 (thêm ngày theo thâm niên)
```

**Tái sử dụng policy:**
```
ANNUAL_STANDARD  ─┐
                   ├── ACCRUAL_MONTHLY (1 policy, reused by 2 classes)
ANNUAL_SENIOR_5Y ─┘

ANNUAL_SENIOR_5Y: CARRY_10DAYS (10 ngày thay vì 5 ngày)
```

## 3.3.1 LeavePolicy Entity

### Định danh & Type

| Field | Type | Nullable | Mô tả |
|-------|------|----------|-------|
| `id` | UUID | NOT NULL | PK |
| `tenant_id` | UUID | NULLABLE | Multi-tenancy isolation |
| `code` | varchar(50) | NOT NULL, UNIQUE | e.g., `ACCRUAL_MONTHLY_1D`, `CARRY_5DAYS_POL` |
| `name` | varchar(150) | NOT NULL | Human-readable name |
| `policy_type` | varchar(30) | NOT NULL | Discriminator — xem bảng Policy Types |
| `config_json` | jsonb | NOT NULL | Schema phụ thuộc `policy_type` — xem chi tiết từng type |
| `is_active` | boolean | default: true | |
| `metadata` | jsonb | NULLABLE | |
| `created_at` | timestamptz | default: now() | |
| `updated_at` | timestamptz | NULLABLE | |

> **Note về deprecated fields:** DBML vẫn giữ các column cũ (`accrual_rule_json`, `carry_rule_json`, `limit_rule_json`, `validation_json`, `rounding_json`, `proration_json`) để backward compatibility với code cũ chưa migrate. Về mặt design, **toàn bộ logic đều đặt trong `config_json`** theo `policy_type` discriminator. Dev implement theo `config_json`, không cần đọc/ghi các deprecated columns.

### Policy Types — 7 loại

#### 1. `ACCRUAL` — Cấp phép định kỳ

Định nghĩa cách balance được tích lũy theo thời gian.

```json
{
  "policy_type": "ACCRUAL",
  "config_json": {
    "method": "MONTHLY_FIXED",
    "rate": 1.0,
    "unit": "DAY",
    "cap_per_period": 12,
    "anniversary_reset": false,
    "proration_join": "FULL_MONTH",
    "proration_exit": "ACTUAL_DAYS"
  }
}
```

| `method` | Mô tả |
|----------|-------|
| `MONTHLY_FIXED` | Cấp cố định mỗi tháng (rate = số ngày/tháng) |
| `ANNUAL_GRANT` | Cấp toàn bộ đầu năm (rate = tổng ngày/năm) |
| `BIWEEKLY` | Cấp 2 tuần/lần |
| `HIRE_ANNIVERSARY` | Cấp vào ngày kỷ niệm hợp đồng |

**VLC note:** Annual Leave thông thường dùng `ANNUAL_GRANT` hoặc `MONTHLY_FIXED` tùy chính sách công ty. BLLĐ không quy định phương thức — chỉ quy định tổng số ngày/năm.

---

#### 2. `CARRY` — Quy tắc mang phép sang kỳ sau

```json
{
  "policy_type": "CARRY",
  "config_json": {
    "max_carry_days": 5,
    "expire_months_after_period_end": 3,
    "expiry_action": "FORFEITURE",
    "carry_calculation": "AVAILABLE_AT_PERIOD_END"
  }
}
```

| Field | Mô tả |
|-------|-------|
| `max_carry_days` | Số ngày tối đa được carry-over. `null` = không giới hạn |
| `expire_months_after_period_end` | Số tháng sau khi kỳ kết thúc thì lot carry-over hết hạn |
| `expiry_action` | `FORFEITURE` (mất) / `CASHOUT` (đổi lương) / `EXTENSION` (gia hạn) |
| `carry_calculation` | `AVAILABLE_AT_PERIOD_END` = tính từ balance còn lại cuối kỳ |

**Ví dụ thực tế:** Annual Leave năm 2025, employee còn 7 ngày cuối năm. Policy `max_carry=5`, `expire_months=3`:
- Carry: 5 ngày → lot mới `lot_kind=CARRY`, `expire_date=2026-03-31`
- Forfeit: 2 ngày → movement `EXPIRE -2`

---

#### 3. `SENIORITY` — Thêm ngày theo thâm niên (Điều 113 BLLĐ)

```json
{
  "policy_type": "SENIORITY",
  "config_json": {
    "base_days": 12,
    "increment_per_5years": 1,
    "max_additional_days": 3,
    "vlc_article": "113",
    "tiers": [
      {"min_years": 0,  "max_years": 4,  "total_days": 12},
      {"min_years": 5,  "max_years": 9,  "total_days": 13},
      {"min_years": 10, "max_years": 14, "total_days": 14},
      {"min_years": 15, "max_years": null, "total_days": 15}
    ]
  }
}
```

**VLC Điều 113:** Điều kiện làm việc nặng nhọc/độc hại được tăng thêm ít nhất 3 ngày thêm vào base. Policy `SENIORITY` resolve tier dựa trên `employment.seniority_years` tại thời điểm GRANT event.

---

#### 4. `LIMIT` — Giới hạn sử dụng

```json
{
  "policy_type": "LIMIT",
  "config_json": {
    "max_per_request": 10,
    "min_per_request": 0.5,
    "max_per_month": null,
    "max_per_year": 20,
    "min_advance_notice_days": 3,
    "consecutive_days_limit": null,
    "requires_certificate_after_days": null
  }
}
```

| Field | Mô tả |
|-------|-------|
| `max_per_request` | Số ngày tối đa mỗi request |
| `min_per_request` | Số ngày tối thiểu (e.g., 0.5 = half-day minimum) |
| `max_per_year` | Tổng ngày tối đa trong năm |
| `min_advance_notice_days` | Tối thiểu bao nhiêu ngày phải submit trước. Override từ `leave_type.min_advance_days` |
| `consecutive_days_limit` | Tối đa bao nhiêu ngày liên tiếp |
| `requires_certificate_after_days` | Sick leave: sau bao nhiêu ngày phải nộp giấy bác sĩ |

---

#### 5. `VALIDATION` — Custom validation rules

```json
{
  "policy_type": "VALIDATION",
  "config_json": {
    "blocked_periods": ["YEAR_END", "PAYROLL_CLOSE"],
    "team_limit_check": true,
    "check_blackout_period": true,
    "probation_eligible": false,
    "gender_restriction": null
  }
}
```

Validation rules được execute khi `leave_request` chuyển từ `DRAFT → SUBMITTED`. Các check này bổ sung cho `leave_blackout_period` table (structural check).

---

#### 6. `PRORATION` — Tính pro-rata khi join/exit mid-year

```json
{
  "policy_type": "PRORATION",
  "config_json": {
    "join_method": "ACTUAL_MONTHS",
    "exit_method": "ACTUAL_MONTHS",
    "rounding": "ROUND_UP_HALF",
    "min_months_eligible": 1
  }
}
```

| `method` | Mô tả |
|---------|-------|
| `ACTUAL_MONTHS` | Tính theo số tháng thực tế làm việc trong kỳ |
| `FULL_MONTH` | Làm từ ngày 1–15 = 0 tháng; từ 16–31 = 1 tháng |
| `NONE` | Không pro-rata — cấp full dù join giữa năm |

---

#### 7. `ROUNDING` — Quy tắc làm tròn balance

```json
{
  "policy_type": "ROUNDING",
  "config_json": {
    "method": "ROUND_HALF",
    "precision": 1,
    "min_increment": 0.5
  }
}
```

## 3.3.2 ClassPolicyAssignment Entity

N:N mapping giữa `leave_class` ↔ `leave_policy`.

### Schema

| Field | Type | Nullable | Mô tả |
|-------|------|----------|-------|
| `id` | UUID | NOT NULL | PK |
| `tenant_id` | UUID | NULLABLE | |
| `class_id` | UUID (FK) | NOT NULL | FK → `absence.leave_class.id` |
| `policy_id` | UUID (FK) | NOT NULL | FK → `absence.leave_policy.id` |
| `priority` | int | default: 10 | Thứ tự evaluate khi nhiều policy cùng type. **Lower = evaluated first** |
| `is_override` | boolean | default: false | `true` = policy này override inherited policy cùng type từ parent class |
| `notes` | text | NULLABLE | Admin notes |
| `effective_start` | date | NOT NULL | |
| `effective_end` | date | NULLABLE | null = hiện tại |
| `is_current_flag` | boolean | default: true | SCD Type 2 — chỉ 1 record `is_current=true` per unique_key |
| `created_at` | timestamptz | | |
| `updated_at` | timestamptz | NULLABLE | |
| `created_by` | UUID | NOT NULL | |

**Unique constraint:**
```sql
UNIQUE (tenant_id, class_id, policy_id, is_current_flag)
```

### Priority ordering — Cách hoạt động

Khi LeaveClass có nhiều policy cùng `policy_type`:
```
ANNUAL_STANDARD → ACCRUAL_MONTHLY_1D (priority=10)
ANNUAL_STANDARD → ACCRUAL_ANNIVERSARY_BONUS (priority=20)
```
→ Event engine evaluate priority 10 trước, 20 sau. Kết quả cộng dồn hoặc override tuỳ `is_override`.

### is_override pattern — BU-level customization

```
BASE:    ANNUAL_STANDARD → CARRY_5DAYS  (is_override=false, priority=10)
OVERRIDE: ANNUAL_STANDARD → CARRY_10DAYS (is_override=true, priority=5, scope=bu_id=XYZ)

→ BU XYZ employees: CARRY_10DAYS thay thế CARRY_5DAYS
→ Các BU khác: CARRY_5DAYS vẫn hiệu lực
```

## Common Gotchas

1. **Nhiều SENIORITY policies trên cùng 1 class:** Nếu cả `pol-003` và `pol-007` đều là `SENIORITY` và gắn vào cùng class — event engine sẽ evaluate theo `priority`. Thường chỉ nên có 1 SENIORITY policy/class. Dùng `is_override=true` nếu cần replace chứ đừng stack 2 cái.

2. **SCD Type 2 với `is_current_flag`:** Khi update policy assignment (e.g., tăng carry từ 5 → 10 ngày từ 01/07), KHÔNG update record cũ. Thay vào đó:
   - Set `effective_end = 2024-06-30`, `is_current_flag = false` trên record cũ
   - Insert record mới với `effective_start = 2024-07-01`, `is_current_flag = true`

3. **`config_json` schema validation:** DBML không enforce JSON schema — application layer phải validate `config_json` theo `policy_type` khi save. Recommend JSON Schema definition per policy_type.

4. **Deprecated columns (backward compat):** Nếu thấy `accrual_rule_json`, `carry_rule_json` etc. trong DB — đó là migrated data từ v4. Đọc từ `config_json` theo `policy_type`; ignore deprecated fields.
