# 3.5 · LeaveInstantDetail — FEFO Lots (Grant Tracking)

> **Bounded context:** Absence · **Layer:** Operational (Balance Detail)  
> **DBML table:** `absence.leave_instant_detail`  
> **Upstream:** `absence.leave_instant`  
> **Downstream:** `absence.leave_movement.lot_id`, `absence.leave_reservation_line.source_lot_id`

---

## Mục đích nghiệp vụ

`LeaveInstantDetail` theo dõi từng **lô phép** (lot/grant) trong balance của một nhân viên. Trong khi `LeaveInstant` chỉ cho biết **tổng số** ngày hiện có, `LeaveInstantDetail` cho biết **từng lô** đó đến từ đâu, khi nào hết hạn, và đã dùng bao nhiêu.

**Tại sao cần lot tracking?**

```
Không có lot tracking:
  Balance = 10 ngày. Dùng 3 ngày → còn 7 ngày.
  Nhưng: 3 ngày carry-over từ 2025 (hết hạn 31/3/2026) hay 3 ngày grant năm 2026?
  → Không biết → không enforce FEFO → carry-over hết hạn mà không biết

Có lot tracking (FEFO):
  Lot A: CARRY, 3 ngày, expire 31/3/2026
  Lot B: GRANT, 12 ngày, expire 31/12/2026
  → Dùng 3 ngày: trừ Lot A trước (expire sớm nhất) → FEFO
  → Lot A hết, balance còn Lot B: 12 ngày
```

## Lot Kind — Nguồn gốc lô phép

| `lot_kind` | Mô tả | Tạo khi nào |
|-----------|-------|------------|
| `GRANT` | Grant chuẩn hàng năm (hoặc monthly accrual) | ACCRUAL event chạy (EARN movement) |
| `CARRY` | Carry-over từ kỳ trước | CARRY event cuối kỳ |
| `BONUS` | Thưởng thêm ngày nghỉ từ HR | Manual HR action (ADJUST movement) |
| `TRANSFER` | Chuyển từ class khác (e.g., comp-time → annual | Manual/automated bridge (EARN movement) |
| `OTHER` | Trường hợp đặc biệt khác | Manual |

## FEFO Algorithm — First-Expired-First-Out

### Bước 1: Khi nhân viên submit request (RESERVE)

```python
# Pseudo-code: FEFO lot selection
def select_lots_fefo(instant_id, reserve_qty):
    lots = query("""
        SELECT id, lot_qty, used_qty, expire_date, priority
        FROM absence.leave_instant_detail
        WHERE instant_id = :instant_id
          AND (expire_date IS NULL OR expire_date >= today())
          AND (lot_qty - used_qty) > 0
        ORDER BY priority ASC, expire_date ASC NULLS LAST
    """)

    result = []
    remaining = reserve_qty

    for lot in lots:
        available = lot.lot_qty - lot.used_qty
        take = min(available, remaining)
        result.append({"lot_id": lot.id, "amount": take, "expiry": lot.expire_date})
        remaining -= take
        if remaining <= 0:
            break

    if remaining > 0:
        raise InsufficientBalanceError(f"Short by {remaining} days")

    return result  # → Create LeaveReservationLine per lot
```

### Ví dụ FEFO đầy đủ

**Setup:** Employee Trần Thị B có ANNUAL_STANDARD class, 2 lots:

| Lot | Kind | lot_qty | used_qty | expire_date | priority |
|-----|------|---------|----------|-------------|----------|
| Lot A | CARRY | 3.0 | 0.0 | 2026-03-31 | 5 |
| Lot B | GRANT | 12.0 | 0.0 | 2026-12-31 | 10 |

**Request: Nghỉ phép 4 ngày (2026-02-10 đến 2026-02-13)**

FEFO engine chạy:
1. Ưu tiên Lot A (priority=5, expire=31/3/2026): take 3 ngày → Lot A `used_qty=3.0`
2. Lot A hết → sang Lot B (priority=10): take 1 ngày → Lot B `used_qty=1.0`

```
LeaveReservationLine 1: lot_id=Lot-A, amount=3.0, expiry=2026-03-31
LeaveReservationLine 2: lot_id=Lot-B, amount=1.0, expiry=2026-12-31
```

LeaveInstant sau reserve:
- `current_qty = 15.0` (unchanged — reserve doesn't reduce current)
- `hold_qty = 4.0` (blocked)
- `available_qty = 11.0`

## Lot Creation — Khi nào tạo lot mới?

```
1. EARN movement (ACCRUAL event):
   → INSERT leave_instant_detail(lot_kind=GRANT, eff_date=today, expire_date=period_end)

2. CARRY event (cuối kỳ):
   → INSERT leave_instant_detail(lot_kind=CARRY, expire_date=period_end + carry_expire_months)

3. HR Manual Adjust (BONUS):
   → INSERT leave_instant_detail(lot_kind=BONUS, expire_date=specified_by_hr)

4. OT Comp-Time Bridge:
   → INSERT leave_instant_detail(lot_kind=TRANSFER, tag="OT_COMP_RUN_ID")
```

## Common Gotchas

1. **`used_qty` được update khi nào?**
   - Khi `LeaveReservationLine` được tạo (request SUBMITTED): `used_qty` tăng → nhưng movement source là RESERVE, không phải USE
   - Khi request APPROVED và leave starts: movement USE posted (không tăng thêm used_qty ở detail — chỉ ở movement)
   - Khi request RELEASED (rejected/cancelled): `used_qty` giảm trở lại

   **Thực tế:** `used_qty` trong LeaveInstantDetail = "ngày đang hold HOẶC đã dùng thực sự từ lot này". Cần xem `LeaveMovement` để biết phân biệt RESERVE vs USE.

2. **Lot priority vs expire_date:** Priority là integer human-set. Thường set: CARRY=5, GRANT=10, BONUS=3. Nhưng nếu có 2 GRANT lots từ 2 kỳ khác nhau → phải sort thêm theo `expire_date ASC` để FEFO đúng.

3. **expire_date = null:** Chỉ nên xảy ra với BONUS lot đặc biệt không có expiry. FEFO engine phải xử lý `null expire_date = NULLS LAST` — tức là ưu tiên dùng sau nhất.

4. **LeaveReservationLine chưa được implement bởi dev hiện tại:** Dev hiện tại link `leave_request → leave_reservation` trực tiếp mà không trace lots. Khi dev implement FEFO đầy đủ, `LeaveReservationLine` sẽ được activate — thiết kế đã sẵn sàng trong DBML.
