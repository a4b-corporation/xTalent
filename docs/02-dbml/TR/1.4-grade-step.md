# 1.4 — GradeStep: Bậc lương & Dual Mode

**Schema:** `comp_core`  
**Table:** `comp_core.grade_ladder_step`  
**Tham chiếu DBML:** [`4.TotalReward.V5.dbml`](./4.TotalReward.V5.dbml) — AQ-14 Option D  
**Last Updated:** 2026-04-06

---

## Mục đích nghiệp vụ

`grade_ladder_step` định nghĩa **các bậc (step) trong từng grade của một ladder** — tức là mỗi "ô" trong bảng lương theo bậc. Entity này trả lời câu hỏi: *"Nhân viên ở Grade M1, Bậc 2 thì lương là bao nhiêu?"*

Điểm đặc biệt: entity này hỗ trợ **2 chế độ tính lương** song song (Dual Mode):
- **TABLE_LOOKUP:** giá trị tuyệt đối (HR nhập thủ công)
- **COEFFICIENT_FORMULA:** hệ số × lương cơ sở (theo quy chế hành chính sự nghiệp VN)

---

## Prerequisites

`grade_ladder_step` phụ thuộc chặt vào [1.3-grade-ladder.md](./1.3-grade-ladder.md):

```
grade_ladder  ──(via grade_ladder_grade)──►  grade_v
                                              │
                                              └── grade_ladder_step
                                                   (1 step = 1 bậc trong 1 grade × 1 ladder)
```

> **Orphan Constraint:** `grade_ladder_step(ladder_id, grade_v_id)` chỉ được tạo khi đã có `grade_ladder_grade(ladder_id, grade_v_id)` tương ứng. Application layer MUST enforce.

---

## Dual Mode — Kiến trúc tổng quan

```
grade_ladder_step
  ├── step_amount   decimal  ── chế độ TABLE_LOOKUP
  └── coefficient   decimal  ── chế độ COEFFICIENT_FORMULA
             │
             │  × VN_LUONG_CO_SO (từ PR domain)
             │    pay_master.statutory_rule.code = 'VN_LUONG_CO_SO'
             │
             └── salary (Gross projected)

Mode được chọn bởi:
  pay_master.pay_profile.grade_step_mode (PR domain)
    ├── 'TABLE_LOOKUP'         → dùng step_amount
    └── 'COEFFICIENT_FORMULA'  → dùng coefficient × VN_LUONG_CO_SO
  (chỉ áp dụng khi pay_master.pay_profile.pay_method = 'GRADE_STEP')
```

**Domain Boundary quan trọng:**
```
TR domain (file này):
  step_amount   = giá trị HR-policy projected (Gross)
  coefficient   = hệ số HR-policy projected (Gross)

PR domain (pay_master.statutory_rule):
  VN_LUONG_CO_SO = mức lương cơ sở Nhà nước (hiện tại: 2,340,000 VND/tháng, 2025)
  → PR engine dùng statutory_rule để tính Net (sau PIT, BHXH...)
```

## Schema chi tiết

### Nhóm 1 — Định danh & Vị trí trong Ladder

| Field | Type | Ý nghĩa |
|-------|------|---------|
| `id` | uuid | Primary key |
| `ladder_id` | uuid | FK → `comp_core.grade_ladder`. Supplementary FK (performance queries). |
| `grade_v_id` | uuid | FK → `comp_core.grade_v`. Supplementary FK (performance queries). |
| `step_number` | int | Số bậc trong grade. Bắt đầu từ 1. Ví dụ: Bậc 1, Bậc 2, Bậc 3 |
| `step_name` | varchar(100) | Tên tùy chọn. Ví dụ: "Bậc khởi điểm", "Bậc nâng cao" |

> **FK Note (06Apr2026):** `ladder_id` và `grade_v_id` là **supplementary FKs** — giữ nguyên trong DB cho query performance. Relationship chính thức đi qua `grade_ladder_grade`. Application MUST verify orphan constraint trước khi insert.

---

### Nhóm 2 — TABLE_LOOKUP Mode

| Field | Type | Ý nghĩa |
|-------|------|---------|
| `step_amount` | decimal(18,4) | Giá trị lương tuyệt đối (Gross projected). HR nhập thủ công. Ví dụ: `15,000,000 VND` |
| `currency` | char(3) | Tiền tệ của `step_amount`. Ví dụ: `VND`, `USD` |

**Dùng khi nào:** Pay scale do tổ chức tự định, không theo công thức nhà nước. Phổ biến cho công ty tư nhân, FDI.

---

### Nhóm 3 — COEFFICIENT_FORMULA Mode

| Field | Type | Ý nghĩa |
|-------|------|---------|
| `coefficient` | decimal(8,4) | Hệ số lương. Ví dụ: `2.34`, `3.66`, `6.20` |

**Công thức:**
```
salary_gross = coefficient × VN_LUONG_CO_SO
             = 2.34 × 2,340,000 VND   (lương cơ sở 2025)
             = 5,475,600 VND

Ví dụ ngạch chuyên viên:
  Bậc 1: coefficient = 2.34 → 5,475,600 VND
  Bậc 2: coefficient = 2.67 → 6,247,800 VND
  Bậc 3: coefficient = 3.00 → 7,020,000 VND
  ...
  Bậc 8: coefficient = 4.98 → 11,653,200 VND
```

**Dùng khi nào:** Cơ quan hành chính sự nghiệp nhà nước VN, đơn vị công lập áp dụng thang bảng lương Nhà nước. Mode này giúp khi Nhà nước điều chỉnh `VN_LUONG_CO_SO` → tự động update toàn bộ bậc lương mà không cần nhập lại.

**Cross-reference PR Domain:**
```
pay_master.statutory_rule.code = 'VN_LUONG_CO_SO_2025'
  valid_from: 2025-07-01
  formula_json: {"amount": 2340000, "currency": "VND"}
  rule_category: "GROSS_TO_NET"   (→ PR domain owns this)
```

---

### Nhóm 4 — Step Progression (HR-Policy Rule)

| Field | Type | Ý nghĩa |
|-------|------|---------|
| `months_to_next_step` | smallint | Số tháng tối thiểu ở bậc này trước khi được nâng lên bậc tiếp theo. `NULL` = không có quy tắc tự động. |
| `auto_advance` | boolean | `true` = tự động nâng bậc khi đủ `months_to_next_step`. `false` = chỉ nâng thủ công. |

**Flow khi `auto_advance = true`:**
```
TR Scheduled Job (chạy hàng ngày):
  → Check: months_in_current_step ≥ months_to_next_step
  → If true: tạo compensation_event type = STEP_ADVANCEMENT
  → STEP_ADVANCEMENT event → trigger comp_adjustment proposal
  → comp_adjustment APPROVED → cập nhật compensation.basis
```

**Ví dụ ngạch chuyên viên:**
```
Bậc 1: months_to_next_step=36, auto_advance=true   (3 năm → lên Bậc 2)
Bậc 2: months_to_next_step=36, auto_advance=true   (3 năm → lên Bậc 3)
Bậc 3: months_to_next_step=36, auto_advance=true
...
Bậc 8: months_to_next_step=null, auto_advance=false  (bậc cuối cùng)
```

---

### Nhóm 5 — Temporal

| Field | Type | Ý nghĩa |
|-------|------|---------|
| `effective_start` | date | Ngày bậc này có hiệu lực |
| `effective_end` | date \| null | `NULL` = vẫn đang active |

**Unique Index:** `(ladder_id, grade_v_id, step_number, effective_start)` — cho phép versioning step theo thời gian.

## Chọn Mode nào?

| Mode | Cấu hình tại | Đọc từ | Dùng khi |
|------|-------------|--------|---------|
| TABLE_LOOKUP | `step_amount` | TR trực tiếp | Công ty tư nhân / FDI tự định mức |
| COEFFICIENT_FORMULA | `coefficient` + `VN_LUONG_CO_SO` (PR) | TR × PR data contract | Đơn vị nhà nước / công lập |

Mode được quyết định bởi `pay_master.pay_profile.grade_step_mode` (PR domain):
- Xem `5.Payroll.V4.dbml`, table `pay_master.pay_profile`, field `grade_step_mode`, line ~475
- Chỉ applicable khi `pay_profile.pay_method = 'GRADE_STEP'`
- Các pay_method khác (`MONTHLY_SALARY`, `HOURLY`, ...) → không dùng grade step mode

> **Cả hai fields (`step_amount` và `coefficient`) đều có thể được điền vào cùng 1 record.** Việc mode nào được dùng phụ thuộc vào `pay_profile.grade_step_mode` tại runtime — không phải tại design time của grade step.

## step_amount và BASE Component Convention

`grade_ladder_step.step_amount` là giá trị projected Gross tương đương component `component_type = BASE` (xem [1.2-pay-component.md](./1.2-pay-component.md)):

```
Khi employee được assign:
  grade_v = M1, grade_ladder_step.step_number = 2

→ TABLE_LOOKUP mode:
  step_amount = 15,000,000 VND
  → compensation.basis_line.amount = 15,000,000 VND (cho component BASE)

→ COEFFICIENT_FORMULA mode:
  coefficient = 2.67
  VN_LUONG_CO_SO = 2,340,000 VND
  → projected Gross = 6,247,800 VND
  → compensation.basis_line.amount = 6,247,800 VND (cho component BASE)
```

## Integration Points

| Module | Hướng | Mục đích |
|--------|-------|---------|
| `comp_core.grade_ladder_grade` | → (prerequisite) | Orphan constraint — phải tồn tại trước khi tạo step |
| `pay_master.pay_profile.grade_step_mode` (PR) | → (runtime decision) | Quyết định TABLE_LOOKUP hay COEFFICIENT_FORMULA |
| `pay_master.statutory_rule` (PR) | → (data contract) | `VN_LUONG_CO_SO` cho COEFFICIENT_FORMULA mode |
| `compensation.basis` (Core) | ← (referenced by) | Nhân viên được gán vào step cụ thể |
| `comp_core.comp_adjustment` | ← (STEP_ADVANCEMENT) | Auto-advance tạo adjustment event |

## Common Gotchas

| Gotcha | Cách tránh |
|--------|-----------|
| Tạo step không có `grade_ladder_grade` | Luôn verify orphan constraint trước khi insert |
| Nhầm mode — điền cả `step_amount` lẫn `coefficient` | Cả hai có thể điền — mode được quyết định bởi `pay_profile.grade_step_mode` at runtime |
| `VN_LUONG_CO_SO` outdated | Khi Nhà nước điều chỉnh, update `statutory_rule` ở PR domain; TR coefficient không cần đổi |
| auto_advance nhưng thiếu `months_to_next_step` | Validate: nếu `auto_advance=true` thì `months_to_next_step` MUST NOT NULL |
| Step tồn tại nhưng không có effective overlap | Query luôn kèm `effective_start <= :today AND (effective_end IS NULL OR effective_end >= :today)` |
