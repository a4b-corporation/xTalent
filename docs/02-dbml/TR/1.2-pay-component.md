# 1.2 — PayComponentDef: Định nghĩa thành phần lương

**Schema:** `comp_core`  
**Tables:** `comp_core.pay_component_def` · `comp_core.salary_basis_component_map`  
**Tham chiếu DBML:** [`4.TotalReward.V5.dbml`](./4.TotalReward.V5.dbml) — section `1. Core Compensation`  
**Last Updated:** 2026-04-06

---

## Mục đích nghiệp vụ

`pay_component_def` định nghĩa **các loại thành phần lương** mà hệ thống quản lý (lương cơ bản, phụ cấp, thưởng, khấu trừ...). Mỗi component là một "hạng mục" độc lập trên bảng lương (payslip).

`salary_basis_component_map` là **bảng N:N mapping** — khai báo component nào được gắn vào salary_basis nào, với thứ tự và tùy chỉnh riêng.

**Ví dụ trực quan:**

| Basis | Component gắn vào | Có bắt buộc? |
|---|---|---|
| `MONTHLY_VND_FULL_TIME` | `LUONG_CO_BAN` | ✅ Bắt buộc |
| `MONTHLY_VND_FULL_TIME` | `PHU_CAP_AN_TRUA` | ❌ Tùy chọn |
| `MONTHLY_VND_FULL_TIME` | `PHU_CAP_DIEN_THOAI` | ❌ Tùy chọn |
| `HOURLY_VND_PARTTIME` | (không có component) | — `allow_components=false` |

## Quan hệ với các entity khác

```
pay_component_def
  │
  ├─── salary_basis_component_map ◄── salary_basis (1.1)
  │      (N:N: ai dùng component này?)
  │
  ├─── component_calculation_rule ──► calculation_rule_def (1.6)
  │      (component-level calculation rules)
  │
  ├─── comp_adjustment ◄── comp_core.comp_adjustment
  │      (adjustment chỉ định component nào tăng/giảm)
  │
  └─── employee_comp_snapshot ◄── (bản ghi lương frozen per-employee)
```

## Schema — `pay_component_def`

### Nhóm 1 — Định danh

| Field | Type | Ý nghĩa |
|-------|------|---------|
| `id` | uuid | Primary key |
| `code` | varchar(50) | Mã duy nhất. Ví dụ: `LUONG_CO_BAN`, `PHU_CAP_AN_TRUA`, `THUONG_HIEU_SUAT` |
| `name` | varchar(200) | Tên hiển thị. Ví dụ: "Lương cơ bản", "Phụ cấp ăn trưa" |
| `is_active` | boolean | `true` = đang sử dụng. `false` = đã retire, không dùng cho basis mới |
| `effective_start` | date | Ngày component có hiệu lực |
| `effective_end` | date \| null | `NULL` = vẫn đang active |

---

### Nhóm 2 — Phân loại & Tính toán

| Field | Type | Ý nghĩa |
|-------|------|---------|
| `component_type` | varchar(30) | Loại component — xem bảng taxonomy bên dưới |
| `frequency` | varchar(20) | `HOURLY` \| `MONTHLY` \| `ANNUAL` — tần suất tính component này |
| `calculation_method` | varchar(50) | `FIXED` \| `FORMULA` \| `PERCENTAGE` \| `HOURLY` \| `DAILY` |
| `proration_method` | varchar(30) | `CALENDAR_DAYS` \| `WORKING_DAYS` \| `NONE` — cách prorate khi nghỉ giữa tháng |
| `display_order` | int | Thứ tự hiển thị trên phiếu lương (payslip) |

#### `component_type` Taxonomy

| type | Ý nghĩa | Ví dụ |
|------|---------|-------|
| `BASE` | Lương cơ bản — thành phần bắt buộc | Lương cơ bản, hệ số lương |
| `ALLOWANCE` | Phụ cấp — liên quan công việc hoặc điều kiện | Phụ cấp ăn trưa, phụ cấp điện thoại, phụ cấp xăng xe |
| `BONUS` | Thưởng biến động — gắn với hiệu suất | Thưởng KPI, thưởng 13 tháng, bonus hàng quý |
| `EQUITY` | Quyền chọn cổ phần / RSU | RSU vesting, stock options |
| `DEDUCTION` | Khấu trừ (HR-policy layer) | Khấu trừ nghỉ không phép, hoàn trả đào tạo |
| `OVERTIME` | Lương làm thêm giờ | OT weekday, OT weekend, OT holiday |

> **Domain boundary:** `DEDUCTION` và `OVERTIME` ở đây là **HR-policy layer** (TR domain, Gross). Statutory deductions (PIT, BHXH) thuộc PR domain (`pay_master.statutory_rule`).

#### `calculation_method` Matrix

| method | Cách tính | Dùng khi nào |
|--------|-----------|-------------|
| `FIXED` | Số tiền cố định mỗi kỳ | Lương cơ bản, phụ cấp cố định |
| `PERCENTAGE` | % của một component khác | Thưởng = 15% × lương cơ bản |
| `HOURLY` | Đơn giá × số giờ (từ TA timesheet) | OT, part-time |
| `DAILY` | Đơn giá × số ngày làm | Công nhật |
| `FORMULA` | Logic phức tạp qua `formula_json` | Thưởng theo hiệu suất với nhiều điều kiện |

### Nhóm 3 — Thuế & BHXH

| Field | Type | Ý nghĩa |
|-------|------|---------|
| `taxable` | boolean | `true` = tính vào thu nhập chịu thuế TNCN |
| `tax_treatment` | varchar(30) | `FULLY_TAXABLE` \| `PARTIALLY_EXEMPT` \| `TAX_EXEMPT` |
| `tax_exempt_threshold` | decimal(18,4) | Ngưỡng miễn thuế cho `PARTIALLY_EXEMPT`. Ví dụ: `730,000` VND (phụ cấp ăn trưa theo TT96/2015) |
| `is_subject_to_si` | boolean | `true` = tính vào cơ sở đóng BHXH |
| `si_calculation_basis` | varchar(30) | `FULL_AMOUNT` \| `CAPPED` \| `EXCLUDED` — cách tính BHXH |

**`tax_treatment` — Ví dụ thực tế:**

| Component | tax_treatment | Giải thích |
|-----------|---------------|-----------|
| `LUONG_CO_BAN` | `FULLY_TAXABLE` | Toàn bộ tính vào thu nhập chịu thuế |
| `PHU_CAP_AN_TRUA` | `PARTIALLY_EXEMPT` | ≤ 730,000 VND/tháng miễn thuế (TT96/2015) — phần vượt mới chịu thuế |
| `PHU_CAP_XANG_XE_THEO_QD` | `TAX_EXEMPT` | Phụ cấp đi lại theo chính sách Nhà nước: miễn hoàn toàn |

**`si_calculation_basis` — Ví dụ:**

| Component | si_calculation_basis | Giải thích |
|-----------|---------------------|-----------|
| `LUONG_CO_BAN` | `FULL_AMOUNT` | Toàn bộ tính vào cơ sở đóng BHXH |
| `PHU_CAP_THAM_NIEN` | `CAPPED` | Có trần theo quy định (capped ở mức lương tối đa BHXH) |
| `THUONG_HIEU_SUAT` | `EXCLUDED` | Thưởng KPI không tính BHXH |

> **Note:** Actual tax/SI calculation (tính PIT, BHXH, BHYT) → PR domain (`pay_master.statutory_rule`). TR domain chỉ khai báo metadata (`taxable`, `is_subject_to_si`...) để PR engine biết.

### Nhóm 4 — Proration & Ordering

| Field | Type | Ý nghĩa |
|-------|------|---------|
| `prorated` | boolean | Default proration behavior của component này |
| `proration_method` | varchar(30) | Phương pháp tính proration khi áp dụng |

> **Priority Rule (proration_method):**  
> `salary_basis_component_map.enable_proration` (nếu NOT NULL) **ưu tiên** hơn `pay_component_def.prorated`.  
> Nếu NULL tại mapping → fallback về `pay_component_def.prorated`.

### Nhóm 5 — Scoping

| Field | Type | Ý nghĩa |
|-------|------|---------|
| `country_code` | char(2) | Components là country-level. Ví dụ: `PHU_CAP_AN_TRUA` chỉ áp dụng VN (`country_code=VN`). `NULL` = global component. |
| `config_scope_id` | uuid | Phase 2 scoping (advanced). Xem [1.7-config-scope.md](./1.7-config-scope.md) |

> **Note:** `pay_component_def` không có `legal_entity_id` — components là country-level. LE-specificity được handle thông qua `salary_basis` (xem 1.1).

## Schema — `salary_basis_component_map`

Bảng N:N kết nối `salary_basis` ↔ `pay_component_def`, với tùy chỉnh per-mapping.

| Field | Type | Ý nghĩa |
|-------|------|---------|
| `salary_basis_id` | uuid | FK → `comp_core.salary_basis` |
| `component_id` | uuid | FK → `comp_core.pay_component_def` |
| `mandatory_flag` | boolean | `true` = component bắt buộc, không thể remove khi assign lương cho nhân viên |
| `sort_order` | int | Thứ tự hiển thị component trong context basis này |
| `calculation_order` | int | Thứ tự thực thi trong calculation flow (quan trọng cho inter-component dependency) |
| `display_order` | int | Override `pay_component_def.display_order` cho basis này |
| `enable_proration` | boolean \| null | Override proration setting của component — **NULL = dùng component default** |
| `proration_method` | varchar(30) \| null | Override proration method — NULL = dùng component default |
| `custom_formula_json` | jsonb \| null | Basis-specific formula override — NULL = dùng component formula |

### 3 Loại Ordering

```
sort_order        = thứ tự trong danh sách component của basis này (UI listing)
calculation_order = thứ tự thực thi khi tính toán (logic dependency)
display_order     = thứ tự trên payslip / bảng lương

Ví dụ:
  LUONG_CO_BAN:    sort=1, calc=1, display=1  (tính trước — là base cho % component)
  THUONG_HIEU_SUAT:sort=3, calc=2, display=3  (tính sau, phụ thuộc LUONG_CO_BAN)
  PHU_CAP_AN_TRUA: sort=2, calc=3, display=2  (tính sau cùng vì fixed, không phụ thuộc)
```

### Proration Override Precedence

```
Level 1 (ưu tiên cao nhất):
  salary_basis_component_map.enable_proration IS NOT NULL
  → dùng giá trị này + salary_basis_component_map.proration_method

Level 2 (fallback):
  salary_basis_component_map.enable_proration IS NULL
  → dùng pay_component_def.prorated + pay_component_def.proration_method
```

## Business Rules

### Rule 1: Mandatory Components không thể bỏ khi assign

Khi HR tạo `compensation.basis` cho nhân viên với `salary_basis = MONTHLY_VND_FULL_TIME`, tất cả components có `mandatory_flag=true` phải được gán giá trị. Application layer phải validate.

### Rule 2: calculation_order quyết định thứ tự phụ thuộc

Nếu `THUONG_HIEU_SUAT` = 15% × `LUONG_CO_BAN`, thì `calc_order(LUONG_CO_BAN) < calc_order(THUONG_HIEU_SUAT)`. Engine calculate theo thứ tự tăng dần của `calculation_order`.

### Rule 3: custom_formula_json là basis-specific override

Cùng 1 component `PHU_CAP_DIEN_THOAI` có thể có formula khác nhau giữa `MONTHLY_VND_BASIC` (500K cố định) và `MONTHLY_VND_SENIOR` (1M cố định). Dùng `custom_formula_json` tại mapping.

### Rule 4: step_amount → BASE component convention

`grade_ladder_step.step_amount` là giá trị projected Gross tương đương component `component_type=BASE`. Khi employee được assign grade step, `step_amount` (hoặc `coefficient × VN_LUONG_CO_SO`) trở thành amount của BASE component trong `compensation.basis`.

## Common Gotchas

| Gotcha | Cách tránh |
|--------|-----------|
| Nhầm 3 ordering (sort/calc/display) | `sort_order` = UI list; `calculation_order` = engine sequence; `display_order` = payslip. Kiểm tra cả 3 khi setup |
| Proration apply không đúng | Kiểm tra `enable_proration` ở mapping trước — nếu NOT NULL nó override component default |
| Mandatory component bị thiếu amount | HR form phải validate tất cả `mandatory_flag=true` components có amount trước khi save `compensation.basis` |
| Tax_treatment nhầm | `PARTIALLY_EXEMPT` cần có `tax_exempt_threshold`. Nếu thiếu → engine tính sai phần miễn thuế |
