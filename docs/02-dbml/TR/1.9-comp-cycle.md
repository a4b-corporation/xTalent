# 1.9 — CompCycle: Chu kỳ điều chỉnh lương

**Schema:** `comp_core`  
**Tables:** `comp_core.comp_plan` · `comp_core.comp_cycle` · `comp_core.comp_adjustment` · `comp_core.budget_allocation`  
**Tham chiếu DBML:** [`4.TotalReward.V5.dbml`](./4.TotalReward.V5.dbml) — section `1. Core Compensation`  
**Last Updated:** 2026-04-06

---

## Mục đích nghiệp vụ

Nhóm entity này quản lý **toàn bộ quy trình review và điều chỉnh lương định kỳ** — từ khai báo kế hoạch (Plan), tạo chu kỳ cụ thể (Cycle), đề xuất điều chỉnh từng nhân viên (Adjustment), đến phân bổ ngân sách (Budget).

**Data Lineage:**
```
comp_plan ──creates──► comp_cycle ──contains──► comp_adjustment
                          │                         │
                          └──allocates──► budget_allocation
                                         (track ngân sách)
                                              │
                                              ▼
                                    employee_comp_snapshot (1.10)
                                    (frozen khi adjustment APPROVED)
```

## Schema — `comp_plan`

Kế hoạch điều chỉnh lương — template tái sử dụng cho nhiều năm/chu kỳ.

| Field | Type | Ý nghĩa |
|-------|------|---------|
| `id` | uuid | Primary key |
| `code` | varchar(50) | Mã plan. Ví dụ: `MERIT_VN_2025`, `PROMOTION_GLOBAL` |
| `name` | varchar(200) | Tên. Ví dụ: "Kế hoạch tăng lương theo hiệu suất VN 2025" |
| `plan_type` | varchar(30) | Loại kế hoạch — xem taxonomy |
| `eligibility_profile_id` | uuid | FK → `eligibility.eligibility_profile`. WHO trong plan này. `NULL` = tất cả. |
| `eligibility_rule` | jsonb | **DEPRECATED** — dùng `eligibility_profile_id` thay thế |
| `guideline_json` | jsonb | Merit matrix, approval thresholds, guidelines |
| `country_code` | char(2) | Scoping UI filter. `NULL` = global |
| `legal_entity_id` | uuid | Scoping UI filter. `NULL` = all LEs |
| `config_scope_id` | uuid | Phase 2 scope group |
| `is_active` | boolean | `false` = plan đã retire |
| `effective_start` | date | Ngày plan có hiệu lực |
| `effective_end` | date \| null | `NULL` = active |

### `plan_type` Taxonomy

| type | Ý nghĩa | Tần suất |
|------|---------|---------|
| `MERIT` | Tăng lương theo hiệu suất / đánh giá KPI | Hàng năm |
| `PROMOTION` | Tăng lương theo thăng chức (grade change) | Ad-hoc |
| `MARKET_ADJUSTMENT` | Điều chỉnh theo thị trường / market survey | 1–2 lần/năm |
| `NEW_HIRE` | Offer lương cho nhân viên mới | Ad-hoc |
| `EQUITY_CORRECTION` | Điều chỉnh công bằng nội bộ (pay equity review) | Hàng năm |
| `AD_HOC` | Điều chỉnh ngoài kế hoạch | Bất kỳ lúc nào |

### `guideline_json` Structure

```json
{
  "merit_matrix": {
    "rows": ["BELOW_EXPECTATIONS", "MEETS_EXPECTATIONS", "EXCEEDS_EXPECTATIONS"],
    "cols": ["RED_ZONE", "COMPETITIVE", "ABOVE_MARKET"],
    "values": {
      "BELOW_EXPECTATIONS|RED_ZONE": 0,
      "MEETS_EXPECTATIONS|COMPETITIVE": 4.5,
      "EXCEEDS_EXPECTATIONS|RED_ZONE": 8.0
    }
  },
  "approval_thresholds": {
    "direct_manager": {"max_increase_pct": 5.0},
    "vp": {"max_increase_pct": 10.0},
    "cfo": {"max_increase_pct": 999}
  },
  "budget_cap_pct": 3.5
}
```

## Schema — `comp_cycle`

Một lần thực thi cụ thể của `comp_plan` — ví dụ "Merit 2025 Q1".

| Field | Type | Ý nghĩa |
|-------|------|---------|
| `id` | uuid | Primary key |
| `plan_id` | uuid | FK → `comp_core.comp_plan` |
| `code` | varchar(50) | Mã cycle. Ví dụ: `MERIT_VN_2025_Q1` |
| `name` | varchar(200) | Tên cycle |
| `period_start` | date | Bắt đầu review period |
| `period_end` | date | Kết thúc review period |
| `effective_date` | date | Ngày điều chỉnh có hiệu lực (thường sau `period_end`) |
| `budget_amount` | decimal(18,4) | Tổng ngân sách toàn chu kỳ |
| `budget_currency` | char(3) | Tiền tệ ngân sách |
| `status` | varchar(20) | State machine — xem bên dưới |
| `closed_date` | timestamp | Khi cycle đóng lại |
| `closed_by` | uuid | Ai đóng cycle |

### Comp Cycle State Machine

```
DRAFT ──────────────────────────────────────► CANCELLED
  │
  ▼
OPEN  (HR mở để managers nhập proposals)
  │
  ▼
IN_REVIEW  (submissions đã xong, đang review tổng thể)
  │
  ▼
APPROVED  (C-Suite / CFO sign-off)
  │
  ▼
CLOSED  (adjustments đã apply vào employee_comp_snapshot)
```

| status | Ý nghĩa |
|--------|---------|
| `DRAFT` | HR đang cấu hình, chưa mở nhập liệu |
| `OPEN` | Managers đang tạo `comp_adjustment` proposals |
| `IN_REVIEW` | Submission deadline qua, HR/Finance review tổng |
| `APPROVED` | C-Suite approve tổng thể |
| `CLOSED` | Đã apply adjustments → tạo `employee_comp_snapshot` |
| `CANCELLED` | Cycle bị hủy trước khi hoàn tất |

## Schema — `comp_adjustment`

Đề xuất điều chỉnh lương cho từng nhân viên trong một cycle.

| Field | Type | Ý nghĩa |
|-------|------|---------|
| `id` | uuid | Primary key |
| `cycle_id` | uuid | FK → `comp_core.comp_cycle` |
| `employee_id` | uuid | FK → `employment.employee` |
| `assignment_id` | uuid | FK → `employment.assignment` (grade context) |
| `component_id` | uuid | FK → `comp_core.pay_component_def` (component nào được điều chỉnh) |
| `current_amount` | decimal(18,4) | Lương hiện tại của component này |
| `proposed_amount` | decimal(18,4) | Đề xuất mới |
| `increase_amount` | decimal(18,4) | Computed: `proposed - current` |
| `increase_pct` | decimal(7,4) | Computed: `(proposed - current) / current × 100` |
| `currency` | char(3) | Tiền tệ |
| `rationale_code` | varchar(50) | Mã lý do chuẩn hóa |
| `rationale_text` | text | Giải thích thêm từ manager |
| `approval_status` | varchar(20) | State machine — xem bên dưới |
| `submitted_date` | timestamp | Khi manager submit |
| `submitted_by` | uuid | Manager nào submit |
| `approver_emp_id` | uuid | Approver cuối cùng (final decision) |
| `decision_date` | timestamp | Khi có quyết định |
| `decision_note` | text | Ghi chú của approver |

### CompAdjustment Approval State Machine

```
DRAFT ──────────────────────────────────────► WITHDRAWN (manager rút lại)
  │
  ▼ (manager submit)
SUBMITTED ──────────────────────────────────► WITHDRAWN
  │
  │ [Temporal Workflow Engine tiếp nhận]
  ▼
PENDING (đang trong approval queue)
  │
  ├──────────────────────────────────────────► REJECTED
  │
  ▼
APPROVED ──────────────────────────────────► (trigger tạo employee_comp_snapshot)
```

| status | Ý nghĩa | Owner |
|--------|---------|-------|
| `DRAFT` | Manager đang soạn thảo | Manager |
| `SUBMITTED` | Manager đã gửi lên | Temporal Engine tiếp nhận |
| `PENDING` | Đang trong approval queue của approver | Approver |
| `APPROVED` | Được duyệt → trigger employee_comp_snapshot | System |
| `REJECTED` | Bị từ chối với `decision_note` | Approver |
| `WITHDRAWN` | Manager rút lại trước khi approved | Manager |

**Workflow Note:** Chi tiết approval chain (ai approve level nào, SLA, escalation) được quản lý bởi **Temporal Workflow Engine** — NOT stored in DB. DB chỉ lưu final outcome: `approval_status` + `decision_note` + `decision_date`.

## Schema — `budget_allocation`

Phân bổ và track ngân sách điều chỉnh lương theo scope tổ chức.

| Field | Type | Ý nghĩa |
|-------|------|---------|
| `id` | uuid | Primary key |
| `cycle_id` | uuid | FK → `comp_core.comp_cycle` |
| `scope_type` | varchar(30) | `GLOBAL` \| `LEGAL_ENTITY` \| `BUSINESS_UNIT` \| `DEPARTMENT` \| `TEAM` |
| `scope_uuid` | uuid | Dynamic FK — xem resolution table |
| `allocated_amount` | decimal(18,4) | Ngân sách được phân bổ |
| `utilized_amount` | decimal(18,4) | Đã sử dụng (tổng proposed_amount của APPROVED adjustments) |
| `utilization_pct` | decimal(5,2) | Computed: `utilized / allocated × 100` |
| `currency` | char(3) | Tiền tệ ngân sách |

**Scope Resolution Table:**

| scope_type | scope_uuid trỏ về |
|------------|------------------|
| `GLOBAL` | `NULL` |
| `LEGAL_ENTITY` | `org_legal.entity.id` |
| `BUSINESS_UNIT` | `org.business_unit.id` |
| `DEPARTMENT` | `org.department.id` |
| `TEAM` | `org.team.id` |

**Budget Tracking Flow:**
```
1. HR setup: budget_allocation(cycle, LEGAL_ENTITY, uuid-vn-le, allocated=1,000,000,000 VND)
2. Managers tạo comp_adjustments
3. Khi adjustment APPROVED:
   → budget_allocation.utilized_amount += adjustment.increase_amount
   → utilization_pct recomputed
4. HR theo dõi: utilization_pct > 100% → ngân sách vượt → alert
```

## End-to-End Flow

```
1. HR Admin tạo comp_plan (MERIT_VN_2025)
   → guideline_json: merit matrix, approval thresholds
   → eligibility_profile_id: VN full-time employees

2. HR Admin tạo comp_cycle (MERIT_VN_2025_Q1)
   → plan_id, period_start=2025-01-01, period_end=2025-01-31
   → effective_date=2025-02-01 (lương mới apply từ tháng 2)
   → status=DRAFT

3. HR setup budget_allocation per LE/BU
   → VN_LE: allocated=1,000,000,000 VND

4. HR set cycle status=OPEN → Managers bắt đầu nhập

5. Managers soạn comp_adjustment (status=DRAFT)
   → current=20M, proposed=21M, increase_pct=5%
   → Managers submit (status=SUBMITTED)

6. Temporal Engine: routing theo guideline_json thresholds
   → increase_pct=5% ≤ 5% → direct_manager approve đủ
   → Manager approves → status=APPROVED

7. System: budget_allocation.utilized_amount += 1,000,000 VND

8. HR: IN_REVIEW → APPROVED (tổng thể) → CLOSED

9. Post-cycle: System tạo employee_comp_snapshot (status=ACTIVE)
   → effective_start = cycle.effective_date = 2025-02-01
   → source_type=COMP_ADJUSTMENT, source_ref=adjustment.id
```

## Integration Points

| Module | Hướng | Mục đích |
|--------|-------|---------|
| `eligibility.eligibility_profile` | → (FK) | WHO được include trong comp_plan |
| `employment.employee / assignment` | → (FK) | Nhân viên và assignment context |

## Common Gotchas

| Gotcha | Cách tránh |
|--------|-----------|
| Budget over-allocation | Monitor `utilization_pct` real-time; alert khi > 95% |
| APPROVED adjustment không tạo snapshot | Verify trigger/event từ approval flow đến snapshot creation |
| comp_cycle status skip | Enforce state machine — không cho phép jump từ DRAFT → APPROVED |
| Nhiều adjustment cho cùng employee×component | Unique constraint: `(cycle_id, employee_id, component_id)` nên được enforce |
