# 1.10 — EmployeeCompSnapshot: Bản ghi lương nhân viên

**Schema:** `comp_core`  
**Table:** `comp_core.employee_comp_snapshot`  
**Tham chiếu DBML:** [`4.TotalReward.V5.dbml`](./4.TotalReward.V5.dbml) — section `1. Core Compensation`  
**Last Updated:** 2026-04-06

---

## Mục đích nghiệp vụ

`employee_comp_snapshot` là **bản ghi lương frozen per-employee** — lưu giá trị lương cụ thể của từng thành phần lương cho từng nhân viên tại các thời điểm cụ thể. Đây là "đĩa cơm đã bưng ra bàn" — immutable audit record, phân biệt với configuration.

**Phân biệt hai entity quan trọng:**

| | `salary_basis` (1.1) | `employee_comp_snapshot` |
|---|---|---|
| **Bản chất** | Config template (Master Data) | Frozen record per-employee |
| **Chứa** | Cấu trúc + loại lương | Số tiền thực tế |
| **Ai có** | Tất cả member của eligibility group | Cụ thể từng nhân viên |
| **Thay đổi** | Khi admin thay đổi policy | Khi có comp_adjustment APPROVED |
| **Dùng bởi** | HR Admin config | Payroll engine, reports |

---

## Nguồn tạo Snapshot (source_type)

```
source_type = COMP_ADJUSTMENT  ← từ comp_adjustment APPROVED (1.9)
source_type = OFFER_PACKAGE    ← từ tr_offer.offer_package (khi new hire accept offer)
source_type = MASS_UPLOAD      ← từ bulk upload (migration, mass update)
source_type = MANUAL           ← HR nhập tay (trường hợp đặc biệt)
```

## Schema chi tiết

| Field | Type | Ý nghĩa |
|-------|------|---------|
| `id` | uuid | Primary key |
| `employee_id` | uuid | FK → `employment.employee` |
| `assignment_id` | uuid | FK → `employment.assignment` (multi-assignment support) |
| `component_id` | uuid | FK → `comp_core.pay_component_def` — component nào |
| `amount` | decimal(18,4) | Giá trị lương của component này |
| `currency` | char(3) | ISO 4217 |
| `frequency` | varchar(20) | `HOURLY` \| `MONTHLY` \| `ANNUAL` — frequency của amount này |
| `status` | varchar(20) | `PLANNED` \| `ACTIVE` \| `EXPIRED` — lifecycle state |
| `source_type` | varchar(30) | Nguồn tạo snapshot |
| `source_ref` | varchar(100) | ID của record nguồn (ví dụ: `comp_adjustment.id`) |
| `effective_start` | date | Ngày amount này có hiệu lực |
| `effective_end` | date \| null | `NULL` = vẫn đang ACTIVE |
| `metadata` | jsonb | Extensible extra info |

## Status Lifecycle

```
PLANNED ──────────────────────────────────────► (effective_start đến)
  │                                              │
  │  (khi effective_start = today)              │
  ▼                                             │
ACTIVE ◄─────────────────────────────────────── ┘
  │
  │  (khi có snapshot mới hoặc explicit expire)
  ▼
EXPIRED
```

| status | Ý nghĩa | Trigger |
|--------|---------|---------|
| `PLANNED` | Đã tạo nhưng chưa đến ngày hiệu lực | `effective_start` > today |
| `ACTIVE` | Đang áp dụng — lương hiện tại của nhân viên | `effective_start` ≤ today AND `effective_end` IS NULL |
| `EXPIRED` | Đã thay thế bởi snapshot mới | Khi tạo snapshot mới → snapshot cũ đặt `effective_end + status=EXPIRED` |

**Thực tế query ACTIVE:**
```sql
SELECT * FROM comp_core.employee_comp_snapshot
WHERE employee_id = :emp_id
  AND status = 'ACTIVE'
  -- Index: (employee_id, status, effective_start) WHERE status = 'ACTIVE'
```

## Temporal Lifecycle — Ví dụ

```
Nhân viên Nguyễn Văn A — LUONG_CO_BAN:

2025-01-01: Onboard — Offer Package accepted
  snapshot_1: amount=18,000,000 VND, status=ACTIVE
              source_type=OFFER_PACKAGE, effective_start=2025-01-01

2025-02-01: Merit Cycle MERIT_VN_2025_Q1 → +5%
  snapshot_2: amount=18,900,000 VND, status=ACTIVE
              source_type=COMP_ADJUSTMENT, source_ref=uuid-adj-01
              effective_start=2025-02-01

  → snapshot_1: status=EXPIRED, effective_end=2025-01-31

2025-07-01: Promotion → thêm component PHU_CAP_CHUC_VU
  snapshot_3: component=PHU_CAP_CHUC_VU, amount=3,000,000 VND, status=ACTIVE
              source_type=COMP_ADJUSTMENT, effective_start=2025-07-01
  (snapshot_2 vẫn ACTIVE cho LUONG_CO_BAN)

Lương tháng 08/2025:
  LUONG_CO_BAN:    18,900,000 (snapshot_2, ACTIVE)
  PHU_CAP_CHUC_VU:  3,000,000 (snapshot_3, ACTIVE)
  Total Gross:     21,900,000 → gửi PR engine → tính Net
```

## Frequency Mismatch Handling

Khi `employee_comp_snapshot.frequency` ≠ payroll cycle frequency → cần annualization/de-annualization:

```
Ví dụ:
  snapshot.frequency = ANNUAL, amount = 240,000,000 VND
  payroll cycle = MONTHLY

  → ANNUALIZATION rule (calculation_rule_def):
    monthly_gross = annual_amount / 12 = 20,000,000 VND/tháng
```

Áp dụng `ANNUALIZATION` calculation rule (xem [1.6-calculation-rule.md](./1.6-calculation-rule.md)).

## Relationship với `compensation.basis` (Core/Employment)

```
employment module:
  compensation.basis ──────────────────────────────────────────────────────────►
    (current salary assignment — active config per employee)
    fields: salary_basis_id, employee_id, effective_start, is_current_flag

comp_core:
  employee_comp_snapshot ──────────────────────────────────────────────────────►
    (frozen audit record — immutable after creation)
    fields: employee_id, component_id, amount, status=ACTIVE

Relationship:
  compensation.basis → defines WHICH salary_basis (cấu trúc)
  employee_comp_snapshot → records ACTUAL AMOUNTS (số tiền)
  Payroll reads: compensation.basis (context) + employee_comp_snapshot (amounts)
```

## Payroll Integration

```
PR engine khi tính payroll cho cycle [2025-02]:
  1. SELECT emp_id FROM pay_engine.run_employee WHERE batch_id = :batch
  2. For each employee:
     SELECT * FROM comp_core.employee_comp_snapshot
     WHERE employee_id = :emp_id AND status = 'ACTIVE'
       AND effective_start <= period_end
       AND (effective_end IS NULL OR effective_end >= period_start)
     → list of components + amounts (Gross input)
  3. Apply TR calculation rules (PRORATION for mid-month joiners)
  4. Send Gross → PR: BHXH / BHYT / BHTN / PIT → Net
  5. Write pay_engine.result
```

## Business Rules

### Rule 1: Immutability — Không sửa snapshot sau khi tạo

`employee_comp_snapshot` là immutable audit record. Khi cần sửa:
- Đặt `effective_end = today - 1`, `status = EXPIRED` cho snapshot hiện tại
- Tạo snapshot mới với giá trị đúng

### Rule 2: Expire snapshot cũ khi tạo snapshot mới

Khi `comp_adjustment` APPROVED và tạo snapshot mới:
1. Find existing ACTIVE snapshot cho cùng `(employee_id, component_id)`
2. Set `effective_end = new_effective_start - 1 day`
3. Set `status = EXPIRED`
4. Insert snapshot mới với `status = ACTIVE`

### Rule 3: source_ref traceability

`source_ref` phải chứa ID của record nguồn:
- `COMP_ADJUSTMENT` → `comp_adjustment.id`
- `OFFER_PACKAGE` → `tr_offer.offer_package.id`
- `MASS_UPLOAD` → batch upload job ID

### Rule 4: Multi-assignment support

Nhân viên có thể có nhiều assignments (multi-job). `assignment_id` giúp phân biệt snapshot theo assignment context:
```sql
-- Lương cho assignment chính (primary)
SELECT * FROM employee_comp_snapshot
WHERE employee_id = :emp_id AND assignment_id = :primary_assignment_id AND status = 'ACTIVE'
```

## Common Gotchas

| Gotcha | Cách tránh |
|--------|-----------|
| Sửa snapshot trực tiếp thay vì expire + tạo mới | Rule 1: Immutability — luôn expire + insert |
| ACTIVE snapshot overlap (2 records cùng dates) | Unique index: `(employee_id, component_id, effective_start)` |
| Frequency mismatch với PR payroll | Luôn kiểm tra frequency alignment — apply ANNUALIZATION rule nếu cần |
| source_ref null → mất traceability | Mandatory ở application layer; chỉ `MANUAL` mới được có source_ref dạng mô tả |
