# 1.3 — GradeLadder: Thang cấp bậc & Ngạch lương

**Schema:** `comp_core`  
**Tables:** `comp_core.grade_v` · `comp_core.grade_ladder` · `comp_core.grade_ladder_grade`  
**Tham chiếu DBML:** [`4.TotalReward.V5.dbml`](./4.TotalReward.V5.dbml) — section `1. Core Compensation`  
**Last Updated:** 2026-04-06

---

## Mục đích nghiệp vụ

Nhóm entity này xây dựng **hệ thống thang lương theo cấp bậc** (career ladder / pay scale), trả lời câu hỏi:
- *"Các cấp bậc trong tổ chức này là gì?"* → `grade_v`
- *"Cấp bậc được xếp vào lộ trình nghề nghiệp nào?"* → `grade_ladder`
- *"Ngạch Management có những Grade nào, theo thứ tự nào?"* → `grade_ladder_grade`

Bậc lương thực tế (số tiền hoặc hệ số tại từng bậc) nằm ở [1.4-grade-step.md](./1.4-grade-step.md).

**Ví dụ trực quan:**

```
GradeLadder: MANAGEMENT_TRACK       GradeLadder: TECHNICAL_TRACK
  Grade M1 (job_level=5) — Step 1,2,3  Grade T1 (job_level=3) — Step 1,2,3
  Grade M2 (job_level=6) — Step 1,2,3  Grade T2 (job_level=4) — Step 1,2,3
  Grade M3 (job_level=7) — Step 1,2,3  Grade T3 (job_level=5) — Step 1,2,3
  Grade M4 (job_level=8) — Step 1,2,3  Grade T4 (job_level=6) — Step 1,2,3
```

## Quan hệ giữa 3 Entity

```
grade_ladder  ─────────────────────────────────────────────┐
  (MANAGEMENT_TRACK, TECHNICAL_TRACK, SPECIALIST_TRACK)    │
                                                            │ grade_ladder_grade
grade_v  ──────────────────────────────────────────────────┤  (N:N join, sort_order)
  (M1, M2, M3, T1, T2, T3, T4, S1, S2...)                 │
  SCD Type 2 versioned                                      │
                                                            │
                  └───────────────────────────────────────► grade_ladder_grade
                                                             │
                                                             ▼
                                                       grade_ladder_step (1.4)
                                                        (step 1, 2, 3 per grade×ladder)
```

**Orphan Constraint (critical):**
> `grade_ladder_step` cho `(ladder_id, grade_v_id)` chỉ được tạo khi đã có `grade_ladder_grade` record cho cùng `(ladder_id, grade_v_id)`. Application layer phải enforce.

---

## Schema — `grade_v` (versioned Grade)

`grade_v` implement **SCD Type 2** — mỗi thay đổi của Grade tạo ra một version record mới, giữ nguyên lịch sử.

### Fields

| Field | Type | Ý nghĩa |
|-------|------|---------|
| `id` | uuid | Primary key — mỗi version có 1 ID riêng |
| `grade_code` | varchar(20) | Mã Grade. Bất biến qua các version. Ví dụ: `M1`, `T3`, `S2`, `CHUYEN_VIEN` |
| `name` | varchar(100) | Tên hiển thị. Ví dụ: "Chuyên viên Bậc 1", "Senior Engineer", "Manager L3" |
| `description` | text | Mô tả chi tiết về Grade này |
| `job_level` | int | Số nguyên biểu thị cấp độ trong tổ chức. Dùng để sort và so sánh ngang. |
| `sort_order` | int | Thứ tự hiển thị trong UI (khi list grades) |

**`job_level` Convention:**

| job_level | Tier | Ví dụ |
|-----------|------|-------|
| 1–2 | Entry | Fresher, Junior |
| 3–4 | Mid | Mid-level, Specialist |
| 5–6 | Senior | Senior, Lead, Manager |
| 7–8 | Director | Director, VP |
| 9–10 | Executive | SVP, C-Suite |

### SCD Type 2 Fields

| Field | Type | Ý nghĩa |
|-------|------|---------|
| `effective_start` | date | Ngày version này có hiệu lực |
| `effective_end` | date \| null | Ngày version này hết hiệu lực. `NULL` = vẫn đang current |
| `version_number` | int | Tăng dần: 1, 2, 3... |
| `previous_version_id` | uuid | FK về version trước. `NULL` = version đầu tiên |
| `is_current_version` | boolean | `true` = đây là version hiện tại. Unique index enforce chỉ 1 current per grade_code. |

**SCD-2 Version Chain Ví dụ:**

```
grade_code = "M1" có 2 versions:
  Version 1: id=uuid-a, name="Manager L1", job_level=5, is_current=false
             effective_start=2024-01-01, effective_end=2025-01-01
  Version 2: id=uuid-b, name="Manager Level 1", job_level=5, is_current=true
             effective_start=2025-01-01, effective_end=null
             previous_version_id=uuid-a
```

**Unique Index:** `(grade_code) WHERE is_current_version = true` — đảm bảo mỗi grade_code chỉ có 1 current version.

## Schema — `grade_ladder`

| Field | Type | Ý nghĩa |
|-------|------|---------|
| `id` | uuid | Primary key |
| `code` | varchar(50) | Mã ladder. Ví dụ: `MANAGEMENT_TRACK`, `TECHNICAL_TRACK`, `VN_HANH_CHINH_SU_NGHIEP` |
| `name` | varchar(200) | Tên hiển thị |
| `description` | text | Mô tả lộ trình nghề nghiệp |
| `ladder_type` | varchar(50) | `MANAGEMENT` \| `TECHNICAL` \| `SPECIALIST` — xem taxonomy bên dưới |
| `is_active` | boolean | `false` = ladder đã retire, không assign nhân viên mới |
| `effective_start` | date | Ngày ladder có hiệu lực |
| `effective_end` | date \| null | `NULL` = đang active |

### `ladder_type` Taxonomy

| type | Ý nghĩa | Ai dùng |
|------|---------|---------|
| `MANAGEMENT` | Lộ trình quản lý (Manager → Director → VP → EVP) | People managers |
| `TECHNICAL` | Lộ trình kỹ thuật chuyên sâu (Engineer → Senior → Principal → Fellow) | Individual contributors kỹ thuật |
| `SPECIALIST` | Lộ trình chuyên gia phi-kỹ thuật (Analyst → Senior → Principal...) | Finance, HR, Legal, Operations |

> **Rule:** Một nhân viên chỉ theo 1 ladder_type tại 1 thời điểm. Khi chuyển từ TECHNICAL sang MANAGEMENT (promotion into management) → cần tạo event transfer sang ladder mới.

---

## Schema — `grade_ladder_grade`

Bảng N:N kết nối `grade_ladder` ↔ `grade_v`, với thứ tự sắp xếp.

| Field | Type | Ý nghĩa |
|-------|------|---------|
| `id` | uuid | Primary key |
| `ladder_id` | uuid | FK → `comp_core.grade_ladder` |
| `grade_v_id` | uuid | FK → `comp_core.grade_v` (current version id được recommend) |
| `sort_order` | int | Thứ tự của Grade trong Ladder (thấp = entry, cao = senior) |

**Unique Constraint:** `(ladder_id, grade_v_id)` — 1 grade chỉ xuất hiện 1 lần trong 1 ladder.

**Orphan Constraint với grade_ladder_step:**
> Trước khi tạo `grade_ladder_step(ladder_id=X, grade_v_id=Y)`, phải verify rằng `grade_ladder_grade(ladder_id=X, grade_v_id=Y)` đã tồn tại.

## Ví dụ Cấu hình

### Ladder: MANAGEMENT_TRACK

```
grade_ladder:
  code: MANAGEMENT_TRACK
  name: Lộ trình Quản lý
  ladder_type: MANAGEMENT

grade_ladder_grade (sort_order):
  sort_order=1 → grade_v (grade_code=M1, name="Manager L1", job_level=5)
  sort_order=2 → grade_v (grade_code=M2, name="Manager L2", job_level=6)
  sort_order=3 → grade_v (grade_code=M3, name="Senior Manager", job_level=7)
  sort_order=4 → grade_v (grade_code=D1, name="Director", job_level=8)
  sort_order=5 → grade_v (grade_code=VP, name="Vice President", job_level=9)
```

### Ladder: VN_HANH_CHINH_SU_NGHIEP (Ngạch hành chính sự nghiệp VN)

```
grade_ladder:
  code: VN_NGACH_CHUYEN_VIEN
  name: Ngạch chuyên viên (VN Hành chính sự nghiệp)
  ladder_type: SPECIALIST

grade_ladder_grade:
  sort_order=1 → grade_v (grade_code=CHUYEN_VIEN, name="Chuyên viên")
  sort_order=2 → grade_v (grade_code=CHUYEN_VIEN_CHINH, name="Chuyên viên chính")
  sort_order=3 → grade_v (grade_code=CHUYEN_VIEN_CAO_CAP, name="Chuyên viên cao cấp")
```

## Business Rules

### Rule 1: grade_v là SCD Type 2 — không chỉnh sửa current version

Khi cần thay đổi Grade (đổi tên, thay đổi job_level thể hiện):
1. Set `is_current_version = false`, `effective_end = today` cho version hiện tại
2. Tạo version mới với `version_number + 1`, `previous_version_id = old_id`, `is_current_version = true`

### Rule 2: grade_ladder_grade ref về current version

Khi tạo `grade_ladder_grade`, `grade_v_id` nên trỏ về current version của grade đó (version có `is_current_version = true`). Khi Grade có version mới, cần update `grade_ladder_grade.grade_v_id` sang version mới nếu cần.

### Rule 3: Orphan prevention cho grade_ladder_step

```sql
-- Trước khi INSERT grade_ladder_step:
SELECT 1 FROM grade_ladder_grade
WHERE ladder_id = :ladder_id AND grade_v_id = :grade_v_id
-- Nếu không có row → reject, không cho tạo step orphan
```

### Rule 4: Không mix ladder_type trong 1 ladder

`MANAGEMENT_TRACK` không nên chứa `grade_v` thuộc lộ trình TECHNICAL. Tách thành 2 ladders riêng.

## Common Gotchas

| Gotcha | Cách tránh |
|--------|-----------|
| Tạo `grade_ladder_step` mà không có `grade_ladder_grade` | Application MUST validate orphan constraint (xem Rule 3) |
| Nhầm `grade_v.id` với `grade_code` | `grade_code` = code nghiệp vụ (bất biến, across versions); `grade_v.id` = ID của 1 version cụ thể |
| Chỉnh sửa `grade_v` current version thay vì tạo version mới | SCD Type 2: chỉ tạo version mới, không edit record cũ |
| 1 grade xuất hiện 2 lần trong 1 ladder | `(ladder_id, grade_v_id)` unique constraint sẽ reject |
