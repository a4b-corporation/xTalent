# pay_adjust_reason — Lý do Điều chỉnh Lương (Adjustment Reason)

> **Schema:** `pay_master.pay_adjust_reason`
> **DDD Classification:** Value Object (Reference lookup)
> **Changed:** JUL 2025 (initial)

---

## 1. Config những gì?

`pay_adjust_reason` là **lookup table** cung cấp danh sách lý do chuẩn cho manual adjustments, retroactive corrections, và ad-hoc adjustments trong `pay_mgmt.manual_adjust`. Đảm bảo audit trail có meaningful reason thay vì free-text.

### Fields

| Field | Type | Ý nghĩa | Ví dụ |
|-------|------|---------|-------|
| `code` | varchar(50) UNIQUE | Mã lý do | `RETRO_CORRECTION`, `BONUS_ADHOC` |
| `name` | varchar(100) | Tên hiển thị | `Điều chỉnh hồi tố`, `Thưởng ngoài kế hoạch` |
| `adjust_category` | varchar(30) | Nhóm lý do | `CORRECTION`, `BONUS`, `ALLOWANCE`, `RECOVERY`, `STATUTORY_CHANGE` |
| `requires_approval` | boolean | Cần approval workflow không? | `true` với BONUS_ADHOC |
| `description` | text | Mô tả chi tiết | |
| `is_active` | boolean | | `true` |

---

## 2. Enum — `adjust_category`

| Giá trị | Ý nghĩa |
|---------|---------|
| `CORRECTION` | Sửa lỗi tính lương (retro) |
| `BONUS` | Thưởng ad-hoc |
| `ALLOWANCE` | Phụ cấp bổ sung ngoài profile |
| `RECOVERY` | Thu hồi overpayment |
| `STATUTORY_CHANGE` | Thay đổi do pháp luật (backdated) |
| `TERMINATION` | Điều chỉnh liên quan nghỉ việc |

---

## 3. Seed Data chuẩn

| code | name | adjust_category | requires_approval |
|------|------|:---:|:-:|
| `RETRO_CORRECTION` | Điều chỉnh hồi tố | `CORRECTION` | ❌ |
| `DATA_ENTRY_ERROR` | Sửa lỗi nhập liệu | `CORRECTION` | ❌ |
| `BONUS_PERFORMANCE` | Thưởng hiệu suất | `BONUS` | ✅ |
| `BONUS_ADHOC` | Thưởng ngoài kế hoạch | `BONUS` | ✅ |

| `ADVANCE_RECOVERY` | Thu hồi tạm ứng | `RECOVERY` | ❌ |
| `STATUTORY_UPDATE` | Cập nhật pháp luật hồi tố | `STATUTORY_CHANGE` | ✅ |

---

## 4. Design Notes

> [!NOTE]
> **`requires_approval` là config hint.** Application dùng field này để quyết định có trigger approval workflow không khi HR tạo `manual_adjust` với reason này. Không phải DB constraint — logic ở application layer.

> [!NOTE]
> **Không dùng free-text reason trong manual_adjust.** `pay_mgmt.manual_adjust.reason_code` phải FK về table này. Free-text trong `description/notes` field của manual_adjust là acceptable nhưng `reason_code` phải structured.
