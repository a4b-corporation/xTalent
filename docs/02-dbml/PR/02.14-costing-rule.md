# costing_rule — Quy tắc Phân bổ Chi phí (Costing Rule)

> **Schema:** `pay_master.costing_rule`
> **DDD Classification:** Value Object
> **Changed:** JUL 2025 (initial)

---

## 1. Config những gì?

`costing_rule` xác định **cách phân bổ chi phí lương** vào kế toán: mỗi element được phân bổ cho đơn vị nào (employee, department, cost_center, project), theo tỷ lệ nào, vào GL account nào.

> **Dùng khi:** finance cần split chi phí lương theo cost center (ví dụ: 1 nhân viên làm 2 dự án → split 60/40).

### Fields

| Field | Type | Ý nghĩa | Ví dụ |
|-------|------|---------|-------|
| `code` | varchar(50) UNIQUE | Mã rule | `COST_BY_DEPT`, `COST_BY_PROJECT_SPLIT` |
| `name` | varchar(100) | Tên | `Phân bổ theo phòng ban` |
| `costing_level` | varchar(30) NOT NULL | Mức phân bổ | Xem enum |
| `split_method` | varchar(20) NOT NULL | Cách chia tỷ lệ | `EQUAL`, `MANUAL_PCT`, `TIME_BASED` |
| `gl_account_code` | varchar(50) | GL account mặc định | `622001` |
| `cost_center_id` | uuid FK | Cost center cụ thể. null = derive từ employee | FK → `org.cost_center` |
| `allocation_pct` | decimal(5,2) | % phân bổ nếu split | `60.00` |
| `is_active` | boolean | | `true` |

---

## 2. Enum — `costing_level`

| Giá trị | Phân bổ theo | Dùng khi |
|---------|-------------|---------|
| `EMPLOYEE` | Từng người — không split | Phổ biến nhất |
| `DEPARTMENT` | Phòng ban của employee | Chi phí phòng ban |
| `COST_CENTER` | Cost center chỉ định | Multi-CC allocation |
| `PROJECT` | Project code | Project-based billing |

### `split_method`

| Giá trị | Ý nghĩa |
|---------|---------|
| `EQUAL` | Chia đều cho các cost centers |
| `MANUAL_PCT` | % do admin config (`allocation_pct`) |
| `TIME_BASED` | Split theo % thời gian làm việc (từ TA timesheet) |

---

## 3. Business Rules

| BR | Mô tả |
|----|-------|
| **BR-PR-CR01** | Tổng `allocation_pct` của tất cả records cho cùng employee × period phải = 100%. Application validates. |
| **BR-PR-CR02** | `TIME_BASED` split → engine phải pull timesheet data từ TA trước khi costing. Nếu TA data chưa available → block costing step. |
| **BR-PR-CR03** | Costing output được ghi vào `pay_engine.costing` bảng — không trực tiếp vào GL. GL posting là việc của accounting module. |

## 4. Ví dụ

### Ví dụ 1: Simple — phân bổ theo phòng ban

```json
{
  "code": "COST_BY_DEPT_VN",
  "costing_level": "DEPARTMENT",
  "split_method": "EQUAL",
  "gl_account_code": "641",
  "allocation_pct": 100.00
}
```

### Ví dụ 2: Split project 60/40 (kỹ sư làm 2 project)

```json
[
  { "code": "COST_PROJ_A_60", "costing_level": "PROJECT", "split_method": "MANUAL_PCT",
    "cost_center_id": "<PROJECT_A_CC>", "allocation_pct": 60.00 },
  { "code": "COST_PROJ_B_40", "costing_level": "PROJECT", "split_method": "MANUAL_PCT",
    "cost_center_id": "<PROJECT_B_CC>", "allocation_pct": 40.00 }
]
```

---

## 5. Design Notes

> [!NOTE]
> **Costing ≠ GL Posting.** `costing_rule` định nghĩa allocation logic. `pay_engine.costing` lưu calculated amounts. GL posting là bước riêng của accounting system (thường via `pay_gateway` hoặc ERP integration).
