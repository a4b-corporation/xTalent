# worker_rate_override — Override Đơn giá Per-Worker

> **Schema:** `pay_master.worker_rate_override`
> **DDD Classification:** Value Object
> **SCD-2:** Không — versioned qua `effective_start_date / effective_end_date`
> **Changed:** 27Mar2026 (NEW — AQ-12 Hourly Rate Differentiation, Layer 1)

---

## 1. Config những gì?

`worker_rate_override` là **Layer 1** trong 3-layer rate lookup — per-worker exception rates cho những trường hợp đặc biệt (~5% workers). Khi record này tồn tại, nó **ghi đè hoàn toàn** `pay_profile_rate_config` cho worker đó.

> **Nguyên tắc thiết kế:**
> - Override là **tuyệt đối** (absolute rate), không phải relative multiplier → tránh nhầm lẫn khi tính toán
> - Mọi override phải có **governance trail**: `approved_by`, `approved_at`, `reason_code`
> - Chỉ dùng cho exception thực sự: thương lượng cá nhân, công nhân lành nghề cấp cao, cơ chế bảo lưu

### Nhóm 1 — Binding

| Field | Type | Ý nghĩa | Ví dụ |
|-------|------|---------|-------|
| `employee_id` | uuid FK NOT NULL | Worker được override | FK → `employment.employee` |
| `rate_dimension` | varchar(30) NOT NULL | Chiều rate được override | `REGULAR`, `NIGHT`, `OT_WEEKDAY`... |
| `override_rate` | decimal(18,4) NOT NULL | Rate tuyệt đối (VND/giờ) — KHÔNG phải multiplier | `65000.0000` |
| `is_active` | boolean | Có đang dùng không? | `true` |

### Nhóm 2 — Governance

| Field | Type | Ý nghĩa | Ví dụ |
|-------|------|---------|-------|
| `approved_by` | uuid FK NOT NULL | Người phê duyệt override | FK → `employment.employee` (manager/HR) |
| `approved_at` | timestamptz | Thời điểm phê duyệt | `2024-01-15T09:30:00+07:00` |
| `reason_code` | varchar(50) | Lý do override | `SENIOR_SKILL`, `CONTRACT_NEGOTIATION`, `RETENTION` |
| `notes` | text | Ghi chú thêm | `"Thợ hàn cấp độ 3 — thỏa thuận cá nhân 2024"` |

### Nhóm 3 — Thời gian

| Field | Type | Ý nghĩa |
|-------|------|---------|
| `effective_start_date` | date | Ngày override có hiệu lực |
| `effective_end_date` | date | Ngày hết hiệu lực. `NULL` = đang active |

## 2. Business Rules

| BR | Mô tả |
|----|-------|
| **BR-PR-WRO01** | Unique constraint: `(employee_id, rate_dimension)` per active period. Không thể có 2 active overrides cho cùng (worker, dimension). |
| **BR-PR-WRO02** | `override_rate` là giá trị **tuyệt đối** (VND/giờ), không phải multiplier. Lý do: tránh nhầm lẫn khi base rate thay đổi. Ví dụ: nếu dùng multiplier 1.2× và profile rate thay đổi → override tự thay đổi theo, không còn là "negotiated fixed rate" nữa. |
| **BR-PR-WRO03** | `approved_by` và `approved_at` bắt buộc — không được phép tạo override không có approval. Workflow: HR/manager submit → CFO/HRBP approve → record được created. |
| **BR-PR-WRO04** | Khi override hết hiệu lực (`effective_end_date`), engine tự động fallback về `pay_profile_rate_config`. Không cần action. |
| **BR-PR-WRO05** | Override chỉ dùng cho `rate_dimension = REGULAR`. OT dimensions (`OT_WEEKDAY` etc.) thường vẫn dùng statutory multiplier nhân trên REGULAR rate (kể cả khi REGULAR bị override). |
| **BR-PR-WRO06** | `reason_code` bắt buộc. Ít nhất phải thuộc: `SENIOR_SKILL | CONTRACT_NEGOTIATION | RETENTION | REGULATORY_PROTECTION | SPECIAL_ASSIGNMENT`. Free-text không được chấp nhận. |

## 3. Quan hệ với các entity khác

```
worker_rate_override (Layer 1 — highest priority)
    ↑ overrides
pay_profile_rate_config (Layer 2 — default for profile)
    ↑ uses multiplier from
pay_master.statutory_rule (Layer 3 — OT law multipliers)

pay_engine khi tính rate:
  1. Có worker_rate_override cho (emp, dimension)? → dùng ngay
  2. Không? → lấy pay_profile_rate_config của profile
  3. rate_type = MULTIPLIER + statutory_rule? → apply statutory multiplier
```

## 5. Query Patterns thường gặp

```sql
-- Tất cả active overrides — audit report
SELECT e.employee_code, e.first_name, e.last_name,
       wro.rate_dimension, wro.override_rate, wro.reason_code,
       approver.first_name AS approved_by_name,
       wro.approved_at, wro.effective_start_date, wro.effective_end_date
FROM pay_master.worker_rate_override wro
JOIN employment.employee e ON e.id = wro.employee_id
JOIN employment.employee approver ON approver.id = wro.approved_by
WHERE wro.is_active = TRUE
  AND (wro.effective_end_date IS NULL OR wro.effective_end_date >= CURRENT_DATE)
ORDER BY wro.override_rate DESC;

-- Cost impact: tổng premium đang trả (override - profile rate)
SELECT wro.employee_id,
       wro.override_rate - prc.base_rate AS premium_per_hour
FROM pay_master.worker_rate_override wro
JOIN pay_master.pay_profile_map ppm ON ppm.employee_id = wro.employee_id
  AND ppm.is_current_flag = TRUE
JOIN pay_master.pay_profile_rate_config prc
  ON prc.pay_profile_id = ppm.pay_profile_id
  AND prc.rate_dimension = wro.rate_dimension
  AND prc.is_active = TRUE
WHERE wro.is_active = TRUE
  AND wro.rate_dimension = 'REGULAR';

-- Overrides sắp hết hạn (30 ngày tới)
SELECT e.employee_code, wro.rate_dimension, wro.override_rate, wro.effective_end_date
FROM pay_master.worker_rate_override wro
JOIN employment.employee e ON e.id = wro.employee_id
WHERE wro.effective_end_date BETWEEN CURRENT_DATE AND CURRENT_DATE + 30
  AND wro.is_active = TRUE
ORDER BY wro.effective_end_date;
```

## 6. Design Notes

> [!IMPORTANT]
> **Override là absolute rate, KHÔNG phải multiplier.** Design decision từ AQ-12: nếu dùng multiplier, khi admin thay đổi profile base rate → override tự thay đổi unexpected. Absolute rate = "rate này fixed theo thỏa thuận". HR/manager phải manually review khi profile rate thay đổi.

> [!NOTE]
> **~5% workers cần override** là expectation. Nếu override rate vượt quá 20% total workers → signal rằng profile cần được phân chia thêm (tạo sub-profile). Overuse of override = design smell.

> [!CAUTION]
> **Governance bắt buộc.** Override không có `approved_by` + `approved_at` = invalid data. Application phải require approval workflow trước khi create record. Không cho phép HR tự tạo override không qua approval.
